# ADR-0019: Atlas export schema versioning policy

## Status

Accepted

## Context

PRD-001 IS-13, AC-4.3, § 9.6, and DoD-7 commit to a stable, versioned export schema so editors can `diff` two snapshots taken weeks apart and see real content changes — not noise from reordered keys, renamed fields, or silent format drift.

The atlas's headline "diff over time" use case (A-PAIN-2 in PRD-001 § 2; primary success metric S9) collapses if the JSON shape is unstable. An editor running an export today, then again in November, must be able to `git diff` the two files and have the diff surface only **what actually changed in the tenant** — never *"the developer added a new field at position 3 in October."* This is a design constraint, not a stretch goal.

The PRD spells out the field name (`atlas_export_schema_version`), the starting value (`1`), and the bump rules (`§ 9.6`). What it does not specify, and what implementers reading PRD-001 will need to enforce uniformly, is:

1. **Where does the version constant live?** DoD-7 says "exactly one file" — the architect picks the path so the developer doesn't reinvent it.
2. **What goes in CSV and HTML?** The version must be observable in all three formats so a recipient with only a CSV file can still tell which schema generation it was.
3. **What is the *exact* set of changes that does or does not bump the version?** § 9.6 is mostly right but has edge cases (changing a field's type union; making a previously-optional field required; reordering enum values) that need to be settled before the first PR rather than litigated in code review.

Without an architectural commitment now, the v1 schema lands fine, but v2 will inherit drift the moment any developer interprets "additive change" loosely.

## Decision

### Single source of truth

The schema version constant lives in **one** file: `src/atlas/export/schema-version.ts`.

```ts
// src/atlas/export/schema-version.ts
// Bumping this constant requires an ADR superseding ADR-0019.
// See ADR-0019 for the bump rules.
export const ATLAS_EXPORT_SCHEMA_VERSION = 1 as const;
```

All three format adapters (`json.ts`, `csv.ts`, `html.ts`) import from this file. No other file may declare a literal `1` for the schema version. DoD-7's audit is: `grep -r "atlas_export_schema_version" src/atlas/export/ | grep -v schema-version.ts | grep -v test` returns only **importers**, never additional declarations.

### Surface across formats

| Format | Where the version appears |
|--------|---------------------------|
| JSON | Top-level field: `"atlas_export_schema_version": 1` |
| CSV | Header comment line: `# Schema version: 1` (per PRD-001 § 10.2 header block) |
| HTML | `<footer>` line: `<small>Schema version 1 — generated by Component Usage Atlas.</small>` (per PRD-001 § 10.3) |

A diff-tool consumer of any of the three formats can find the version without parsing JSON.

### Bump rules — exhaustive

A change is **non-breaking** (no version bump) **only if** every existing consumer that could read v1 successfully can also read the new output without modification. Concretely, non-breaking changes are:

- **Adding a new optional top-level field** (e.g. introducing `atlas_export.languages_scanned` in v1.x) — readers that don't know the field ignore it.
- **Adding a new optional field on an existing object** (e.g. adding `last_seen_update` to a rendering body) — same justification.
- **Adding a new value to an open-ended string union** — only when the union is documented as open. (Closed unions like `format: 'json' | 'csv' | 'html'` cannot grow without a bump because consumers may switch on them exhaustively.)
- **Tightening a documentation field** (the `# Schema version: 1` comment in CSV becomes `# Schema version: 1.0`, etc.) without changing parser-visible structure.

A change is **breaking** (MAJOR bump, `1 → 2`) **whenever**:

- A field is **removed** or **renamed** (including rename via casing change).
- A field's **type changes** (`string → number`, `string | null → string`, `T[] → T | T[]`, etc.) — even if "almost equivalent."
- A previously-optional field becomes **required**, or a previously-required field becomes optional.
- The **semantics** of a field change (e.g. `total_usages` switches from "distinct pages" to "total page-rendering combinations").
- An **enum value is removed or renamed**.
- The **declared key order** of a top-level object changes (per IS-14 / AC-4.1 / § 10.4 — order is contractual). Reordering top-level metadata fields is a bump even if every key still exists.
- The **sort order** of any array changes (e.g. switching from `rendering_id ASC` to `total_usages DESC`). Per IS-14, sort order is part of the schema contract.
- The **filename convention** changes (per FR-6 / IS-11). Filenames are part of the schema because tooling that ingests snapshots may key off them.

When a breaking change ships, the new version is `2`. There is no `1.1` / `1.2` / minor version axis in v1. If non-breaking changes accumulate to where readers want to assert "I support v1.7+," a `schema_minor: number` companion field MAY be introduced — but its introduction is itself a non-breaking change at the v1 → v1+ boundary. The `schema_minor` axis is reserved, not yet in v1.

### Backwards compatibility

There is **no** versioned reader/parser shipped with the export module in v1. The atlas reads from its own in-memory state, not from saved exports. Editors who consume snapshots externally (in jq, in a script, in a spreadsheet) own their own parser. The schema version is for **them**, not for the atlas itself.

### Constants in tests

Adapter tests assert `expect(output.atlas_export_schema_version).toBe(1)`. These tests **must** import from `schema-version.ts` so a future bump propagates automatically. A literal `1` hardcoded in a test file is a CR-blocker.

### CHANGELOG discipline

Every release that touches the export module re-states the schema version in `CHANGELOG.md`, even if unchanged. A line `Schema: atlas export schema unchanged at v1.` is the explicit affirmation; a release that bumps the schema gets `Schema: atlas export schema bumped to v2 — see ADR-NNNN.` The /document and /ship stages enforce this.

## Consequences

### Easier

- Diff-tool authors (editors writing jq scripts, BI dashboards consuming CSV) have a single field they can pin and a single contract to read.
- A developer on a future PR who is unsure whether a change is breaking can run the bump-rule checklist mechanically. No "I think it's additive" guesses.
- The schema version is observable in every format, so a recipient who's been emailed only a CSV doesn't have to ask back "what version is this?"

### Harder

- The bump rules are stricter than typical semver-for-schemas. Reordering top-level keys is a major bump, which feels heavy — but the diff-stability argument is what the bump rules protect, and a "harmless" reorder breaks every script that key-orders fields. We accept the strictness.
- Releasing a minor improvement (adding a new optional field) means the schema version stays at `1` while the export's behavior visibly changes. CHANGELOG must spell out *what* changed even when the version doesn't tick; otherwise users may assume "v1 is v1, no change here."
- The single-source-of-truth file (`schema-version.ts`) is a low-traffic file. Developers may forget to import from it and re-declare `const VERSION = 1` somewhere local. The DoD-7 grep audit catches this.

### Neutral

- The choice of `1` as the starting value (over `0` or `1.0.0`) follows the principle that snapshot-schema versioning is integer-major-only at this stage. Users see `1`, then `2`, never `1.0.0` or `1.0.1`. If we ever need a minor axis, we add it; we do not pre-commit to an unused axis.
