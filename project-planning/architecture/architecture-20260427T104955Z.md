## 1. Architecture Overview

Component Usage Atlas is a **single Marketplace app** that ships **two iframe entries** — one mounted at `xmc:dashboardblocks` (the Dashboard Widget) and one at `xmc:pages:context-panel` (the Page Context Panel). Both entries load the **same shared scan engine and atlas singleton** but render different surfaces. The scan engine fans out across the active tenant via the `xmc.agent.*` SDK module, builds two indices in memory (`renderingIndex` + `datasourceIndex`), and freezes the atlas object for UI consumption. Nothing is persisted; nothing crosses tabs; nothing leaves the iframe.

**Architectural shape (top-down):**

- **Surface layer** — two minimal iframe entries, each ≤ 30 KB of entry-specific JS. Each entry renders a small bootstrap that mounts the Marketplace `MarketplaceProvider` (per `client.md` § 2 React-context pattern) and the surface-specific React tree.
- **Atlas layer** — a module-level singleton (per ADR-0003 / FR-4.1) that holds the scan engine, scan-state machine, and frozen atlas result. Both iframe entries import the same module; within a single iframe lifetime, only one scan ever runs.
- **SDK adapter layer** — a single `client.ts` module that wraps `ClientSDK.init({ modules: [XMC] })`, exposes typed `query()` helpers, and centralizes the `requireContextId()` guard from `client.md` § 4 / `xmc.md` § 12a.
- **UI primitive layer** — Blok-derived components (per ADR § 11 Decision K) shared across both surfaces: search-first table, drawer, counters, scope picker, loading visualizer, freshness ribbon.

**Architectural posture (per ADR-0002):** Mode A iframe-only; portal-brokered auth; no `experimental_createXMCClient`; no backend; no persistence; static-hostable bundle.

**Architectural anti-shape:** the bundle is **not** a monolith. It is code-split per route — each of the two iframe entries (`/widget` and `/panel`) loads only what its surface renders, plus the shared atlas chunk. There is no path through the code where the panel pulls in widget-only UI or vice versa.

**What this blueprint locks in (key decisions resolved here, full list in § 11):**

1. **Scaffold:** Marketplace Client-Side (Scaffold 2) per `setup/scaffold.md` — produces a Next.js 16 + Radix + Nova preset app with `MarketplaceProvider` already wired. (§ 11 D1)
2. **Bundler/build:** the scaffold's Next.js + Turbopack default; **no Vite**. Code-splitting is per Next.js App Router route. (§ 11 D2)
3. **UI framework:** Blok (Sitecore design system) — `@blok/*` components via the shadcn registry. (§ 11 D3)
4. **State management:** plain module-singleton with a **tiny in-module pub/sub** (no Zustand, no Redux, no Context for atlas state). (§ 11 D4)
5. **Loading visualization:** **branded animation** (Option A from PRD OQ-2) for v1. Generative-art bloom and mini-game stay deferred. (§ 11 D5)
6. **Concurrency cap:** initial 8 (per FR-1.6); fallback exponential backoff with jitter on 429. Concrete numbers in § 5.5. (§ 11 D6)
7. **Blok theming:** registry stylesheet (`theme.json`) + semantic tokens; no copied tokens. (§ 11 D7)
8. **Telemetry pipeline:** structured `console.info` markers + a **module-level event log buffer** (in-memory only) exposable via a debug affordance; no network egress. (§ 11 D8)
9. **Root route:** `app/page.tsx` returns `notFound()` (§ 3d in `extension-routes.md`); panel + widget routes are reachable directly. We accept the 5c-bis trap by documenting the local smoke-test rule. (§ 11 D9)
10. **Per-page-context handling on the panel:** keep the global scan running, swap the per-page view when the user navigates pages (PRD OQ-3 recommendation accepted). (§ 11 D10)

### 2.6 — Routing / iframe layout

The scaffold ships with `<MarketplaceProvider>` wrapped at `app/layout.tsx`. Per `extension-routes.md` § 5c-bis (P-032, Run 6), this means the root `/` route never resolves outside the portal because the provider sits on the loader forever. We accept this trap (§ 11 D9) — the local smoke-test rule is "always hit `/widget` or `/panel` directly, never `/`." The root `app/page.tsx` is `notFound()` so any accidental visit fails loudly.

**Why we don't move the provider:** moving `<MarketplaceProvider>` out of root layout into `app/(marketplace)/layout.tsx` is the cleaner refactor, but it adds ~one half-day of work for zero behavior delta inside the portal. The trap only bites local smoke tests; we document the rule, save the work, and revisit if a third surface is ever added.

### 2.7 — What is explicitly NOT a system component

- **No backend service** (ADR-0002).
- **No IndexedDB / localStorage / sessionStorage / cookies / external KV** (ADR-0003).
- **No second SDK init path** — only the scaffold's `MarketplaceProvider` (`client.md` § 2). Singleton pattern from § 2.3 of `client.md` is **not** added. The atlas singleton uses the React-provided client.
- **No state management library** — Zustand and Redux are explicitly excluded (§ 11 D4).
- **No router beyond Next.js App Router** — there is no SPA-style nested routing. Each surface is a leaf page.
- **No polling / scheduled refresh** — scans are editor-initiated only (ADR-0003).

**Key flows:**

- **First-mount-on-tab (cold):** surface mounts → `WidgetSurface`/`PanelSurface` subscribes to `atlasStore`. Sees state = `idle`. Calls `runScan({ scope })`. Engine transitions to `scanning`, fans out, transitions to `completed`. Surfaces re-render with the new atlas.
- **Re-mount in same iframe (warm):** surface mounts → subscribes → sees state = `completed`. Renders immediately (NFR-1.3).
- **Cross-iframe (widget tab and panel tab open):** each iframe has its own atlas singleton. Two scans run independently. Acceptable per ADR-0004 "Harder" bullet.
- **Cancel:** user clicks Cancel → `abort()` on the bus → all in-flight `pagesGetComponentsOnPage` calls reject → `Promise.allSettled` resolves → `buildIndices` runs against partial → state = `canceled` (with `partial: true` flag on the atlas).
- **Refresh atlas:** explicit user action → `resetAtlas()` keeps the previous result visible while `runScan` runs again; on completion, the new atlas atomically replaces the old.
- **Scope change (collection picker):** same as Refresh atlas, but `runScan` is called with the new `scope` argument. Previous atlas remains visible (per FR-2.5 / AC-3.2).

## 4. Data Model

### 4.1 — Atlas state machine (the one source of truth)

```ts
// core/atlas-store.ts

export type AtlasScope =
  | { kind: 'all-collections' }
  | { kind: 'collection'; collectionId: string };

export type ScanProgress = {
  phase: 'sites' | 'pages' | 'components';
  pagesTotal: number;       // 0 until phase >= 'components'
  pagesCompleted: number;   // monotonically non-decreasing per phase
  startedAt: number;        // performance.now() at scan start
  elapsedMs: number;        // updated periodically (≥ once/200ms)
};

export type AtlasState =
  | { kind: 'idle' }
  | { kind: 'scanning'; progress: ScanProgress; scope: AtlasScope }
  | { kind: 'completed'; atlas: Atlas; scope: AtlasScope }
  | { kind: 'canceled'; partial: Atlas; scope: AtlasScope; canceledAt: number }
  | { kind: 'error'; reason: AtlasErrorReason; scope: AtlasScope };

export type AtlasErrorReason =
  | { kind: 'no-context' }            // requireContextId() threw
  | { kind: 'sites-fetch-failed'; cause: unknown }
  | { kind: 'unknown'; cause: unknown };
```

**State machine transitions (the only allowed paths):**

```
idle ─────────────► scanning ─┬──► completed
                              │
                              ├──► canceled
                              │
                              └──► error

completed ────────► scanning  (Refresh atlas, or scope change)
canceled  ────────► scanning  (Re-scan after cancel)
error     ────────► scanning  (Retry)

(any) ────────────► idle      (only via resetAtlas() on root unmount)
```

Tested explicitly in `core/scan-state-machine.ts` (§ 6 Test strategy → unit).

### 4.2 — Atlas (the frozen result)

```ts
// lib/sdk/types.ts

export type RenderingId = string;       // rendering definition item ID
export type DatasourceId = string;      // content item ID
export type PageId = string;
export type SiteId = string;
export type CollectionId = string;

export type PageRef = {
  pageId: PageId;
  pageName: string;
  sitePath: string;          // route path under the site (from the agent endpoint)
  siteId: SiteId;
  siteName: string;
  language: string;          // resolved per site (IS-13 / FR-1.3)
  collectionId?: CollectionId;
};

export type RenderingUsage = {
  renderingId: RenderingId;
  displayName: string;        // resolution per ADR-0005
  isUnknown: boolean;         // true if no rendering definition reference (ADR-0005)
  pages: PageRef[];           // every page where this rendering is placed
  totalUsages: number;        // pages.length unless one page uses it twice
  datasources: Set<DatasourceId>;  // distinct datasources bound to this rendering
};

export type DatasourceUsage = {
  datasourceId: DatasourceId;
  displayName: string;
  isMissing: boolean;         // true if not resolvable in tenant content tree (AC-2.5)
  pages: PageRef[];           // pages where any rendering binds this datasource
  renderings: Set<RenderingId>;
};

export type Skipped = {
  pageId: PageId;
  reason: 'forbidden' | 'timeout' | 'not_found' | 'network_error' | 'other';
  detail?: string;            // sanitized — no PII
};

export type Atlas = {
  scope: AtlasScope;
  scannedAt: string;          // ISO timestamp at scan-start
  scannedAtMs: number;        // performance.now() snapshot
  isPartial: boolean;         // true iff the atlas came from a cancel
  renderingIndex: ReadonlyMap<RenderingId, RenderingUsage>;
  datasourceIndex: ReadonlyMap<DatasourceId, DatasourceUsage>;
  skipped: ReadonlyArray<Skipped>;
  totals: {
    sites: number;
    pages: number;
    pagesScanned: number;     // pages where pagesGetComponentsOnPage resolved
    pagesSkipped: number;     // skipped.length
    renderings: number;       // distinct
    datasources: number;      // distinct
  };
};
```

**Freezing.** Once the engine constructs the `Atlas`, it is `Object.freeze`d and the contained `Map` instances are wrapped in a `ReadonlyMap` via a tiny utility. UI selectors must not mutate.

### 4.3 — Build pipeline (what `buildIndices` does)

Pure function. Inputs:
- `pageRefs: ReadonlyArray<PageRef>` — all enumerated pages with resolved language + collection.
- `componentResults: ReadonlyArray<PromiseSettledResult<ComponentRecord[]>>` — per-page fan-out result, in **the same order as `pageRefs`** (the pool preserves order).

Output: `{ renderingIndex, datasourceIndex, skipped, totals }`.

Algorithm (deterministic, single pass):

1. Initialize empty `Map<RenderingId, MutableRenderingUsage>`, `Map<DatasourceId, MutableDatasourceUsage>`, `[] skipped`.
2. For each `(pageRef, settled)` pair, indexed by `i`:
   - If `settled.status === 'rejected'`: classify via `classifyError(settled.reason)` → push `{ pageId: pageRef.pageId, reason }` into `skipped`. Continue.
   - Else, for each component in `settled.value`:
     - Resolve `renderingId` per ADR-0005. If absent, use synthetic id `unknown:<page-id>:<placeholder>:<index>` and mark `isUnknown: true`.
     - Append `pageRef` to `renderingIndex.get(renderingId).pages`.
     - If component carries a direct `datasource.itemId` (per ADR-0006 / FR-8.1) — the rendering's `Datasource` property as returned by `pagesGetComponentsOnPage`:
       - Append `pageRef` to `datasourceIndex.get(dsId).pages`.
       - Add `dsId` to `renderingIndex.get(renderingId).datasources`.
       - Add `renderingId` to `datasourceIndex.get(dsId).renderings`.
     - Inherited / personalized / variant / token-resolved bindings are **ignored** (FR-8.2, ADR-0006 forbidden list).
3. Compute totals.
4. Freeze and return.

Display-name collision handling per FR-9 happens at **render time**, not at index-build time — the index keys by ID; the table component computes the disambiguation suffix when it sees two distinct IDs sharing a display name in its visible result set. This avoids re-rendering the index when the search-filter changes.

### 4.4 — `ComponentRecord` (the SDK shape we depend on)

`xmc.agent.pagesGetComponentsOnPage` returns a per-page response containing components for that page. The exact field names come from `@sitecore-marketplace-sdk/xmc` types — Lead Developer should narrow them in `lib/sdk/types.ts`. The fields we depend on (recorded as a contract here so a future SDK version that drops a field surfaces as a typecheck error):

```ts
// lib/sdk/types.ts — local type aligned with Xmapp.* from xmc.md § 11a
export type ComponentRecord = {
  renderingId?: string;        // rendering definition item ID — primary key per ADR-0005
  renderingName?: string;      // display name; fallback chain per ADR-0005 § "Display name resolution order"
  placeholderKey?: string;     // for synthetic IDs in the unknown-rendering case
  datasource?: {
    itemId?: string;           // content item ID — FR-8.1 / ADR-0006
    displayName?: string;
    path?: string;
  };
  parameters?: Record<string, unknown>;  // for the panel parameter tooltip (AC-2.1)
};
```

## 5. API Design (SDK keys + payload shapes)

This section is the developer's reference contract for all XMC calls. Every call site below uses Mode A (`client.query` with `params:` wrapper, double-unwrap response per `client.md` § 8b). No raw GraphQL. No fallback strategies (per `xmc.md` § 12b).

### 5.1 — Application context (the SDK base map, not XMC)

```ts
// in lib/sdk/queries.ts — but actually consumed via useAppContext() from MarketplaceProvider

// The scaffold's MarketplaceProvider already calls this once on mount:
//   client.query('application.context')  → ApplicationContext at result.data
//
// useAppContext() returns ApplicationContext. We then derive contextId via
// requireContextId(appContext) from core/context-resolver.ts.

import type { ApplicationContext } from '@sitecore-marketplace-sdk/client';

export function requireContextId(ctx: ApplicationContext | null): string {
  const id = ctx?.resourceAccess?.[0]?.context?.live
           ?? ctx?.resourceAccess?.[0]?.context?.preview;
  if (!id) {
    throw new AtlasNoContextError(
      'Sitecore context unavailable — app not bound to a resource',
    );
  }
  return id;
}
```

**`.live` vs `.preview` policy:** prefer `.live` (delivery context) for both surfaces. The atlas reads published structure — the questions editors are answering are about what is delivered. Fall back to `.preview` only when `.live` is missing. (Per `xmc.md` § 12a "policy" note. Locked in here so the resolver doesn't drift between surfaces.)

### 5.2 — Sites enumeration

```ts
// querySitesList — wrapper exported from lib/sdk/queries.ts
export async function querySitesList(
  client: ClientSDK,
  contextId: string,
): Promise<readonly SiteSummary[]> {
  const result = await client.query('xmc.agent.sitesGetSitesList', {
    params: { query: { sitecoreContextId: contextId } },
  });
  return result.data?.data ?? [];   // double-unwrap per client.md § 8b
}
```

**Payload contract** (`xmc.md` § 6 — agent endpoint). Each site at minimum exposes: `{ id, name, displayName?, collectionId?, defaultLanguage? }`. The exact shape comes from `Xmapp.*` types — narrow in `lib/sdk/types.ts`.

### 5.3 — Collections (only when scope is `'collection'`)

```ts
export async function queryListCollections(
  client: ClientSDK,
  contextId: string,

  const result = await client.query('xmc.sites.listCollections', {
    params: { query: { sitecoreContextId: contextId } },

```

Used:
- At surface mount, **always** (to populate the scope picker per AC-3.1) — even when scope is `'all-collections'`.
- During scan, **only** when `scope.kind === 'collection'` (to filter sites in `enumerateSites`).

The result is cached for the iframe lifetime in `atlas-store.ts` under a separate key — `collections` is independent of the atlas itself.

### 5.4 — Site default language

```ts
export async function queryRetrieveSite(
  client: ClientSDK,
  contextId: string,
```

### 5.5 — Pages per site

```ts
export async function queryAllPagesBySite(
  client: ClientSDK,
  contextId: string,
  siteName: string,                // path: siteName per xmc.md § 6
  language: string,
  signal: AbortSignal,
): Promise<readonly PageStub[]> {
  // The XMC SDK does not expose AbortSignal at the call level. We honor cancellation
  // by checking `signal.aborted` immediately on resolution and rejecting if set.
  const result = await client.query('xmc.agent.sitesGetAllPagesBySite', {
    params: {
      path: { siteName },
      query: { language, sitecoreContextId: contextId },
    },
  });
  if (signal.aborted) throw new DOMException('aborted', 'AbortError');
  return result.data?.data ?? [];
}
```

**Note on `path: { siteName }`.** Per `xmc.md` § 6, `sitesGetAllPagesBySite` keys by `siteName`, not by `siteId`. The `enumerateSites` step must therefore retain `siteName` from the sites list response. Lead Developer should verify the exact path-param name against the installed SDK types — record in CATALOG.

### 5.6 — Components per page (the fan-out frontier)

```ts
export async function queryComponentsOnPage(
  client: ClientSDK,
  contextId: string,
  pageId: string,
  language: string,
  signal: AbortSignal,
): Promise<readonly ComponentRecord[]> {
  const result = await client.query('xmc.agent.pagesGetComponentsOnPage', {
    params: {
      path: { pageId },
      query: { language, sitecoreContextId: contextId },
    },
  });
  if (signal.aborted) throw new DOMException('aborted', 'AbortError');
  return result.data?.data ?? [];
}
```

**Concurrency target (per § 11 D6):**

- **Initial cap:** `8` concurrent in-flight calls (FR-1.6, NFR-2.2).
- **Backoff trigger:** when `classifyError` returns `'too_many_requests'` (any 429-like signal — confirm against SDK error shape during implementation; classify as `'other'` if 429 is opaque). Pool reduces effective concurrency by 50% (down to a minimum of `2`) and queues the failed page for retry.
- **Retry policy:** exponential backoff with jitter — `delay = base * 2^attempt + random(0..base)`, where `base = 250ms`, `attempt` capped at `4`. After 4 failed retries, classify the page as `skipped` with reason `'other'`.
- **No silent retry loops** (per NFR-3.3). Every retry is logged via `core/telemetry.ts`.
- **Per-page timeout:** 12 seconds per call. On timeout, reject with `'timeout'`. Per-page failures don't fail the scan (FR-1.7 / NFR-3.1).

### 5.7 — Cross-page navigation (drawer click-through)

```ts
// FR-6.1 — open a page in Pages from the drawer
await client.mutate('pages.context', {
  params: { itemId: pageRef.pageId },
});
```

For widget invocations: a `'pages.context'` mutate from the dashboard widget triggers the host to open the Pages editor focused on that item (per `client.md` § 5). Behavior on the panel surface is the same — the host frame replaces the active page.

**No URL construction** — we do not build a Pages edit URL by hand. We use the SDK's `pages.context` mutation as the canonical click-through. This honors the host frame contract per FR-6.1.

### 5.8 — `pages.context` subscription (panel only)

```ts
// Used by PanelSurface to track the active page.
const res = await client.query('pages.context', {
  subscribe: true,
  onSuccess: (data) => {
    // data.pageInfo?.id is the active page ID; refresh per-page view.
  },
});
// res.unsubscribe() on unmount per client.md § 6a.
```

Per ADR-0004 + `client.md` § 7, only the panel extension point (`xmc:pages:context-panel`) supports `pages.context` (subscribable). The widget never subscribes — it operates on tenant scope only. (PRD OQ-3 resolution per § 11 D10: when the user navigates pages, swap the per-page view; do **not** abort the global scan.)

### 5.9 — Contract: every XMC call passes `sitecoreContextId`

Every `lib/sdk/queries.ts` wrapper takes `contextId: string` as an **already-narrowed argument**. The narrow happens once, at scan-engine entry, via `requireContextId(useAppContext())`. There is **no path** in the codebase where a `query('xmc.*', ...)` call site picks `contextId` from an optional source. This is enforced by:

- TypeScript: every wrapper signature requires `contextId: string` (not `string | undefined`).
- Code review: § 4c-3 of the task breakdown will spell this out as a hard rule.
- Lint: a custom ESLint rule is **not** added — the type system is sufficient.

**No `as any`. No `as never`.** Per `client.md` § 8a / `xmc.md` § 12g.

### 7.2 — Why module-singleton-plus-pub/sub for state (§ 11 D4 deep rationale)

The atlas has the following state-management properties:
- One owner per iframe (the engine).
- Many readers (table, drawer, counters, freshness ribbon, scope picker, loading visualizer).
- Writes are coarse — state changes happen on the order of dozens per scan (every progress-tick batched), not thousands per second.
- State must survive React strict-mode double-mount, hot-reload, and route-internal navigation within the same iframe.
- State must NOT leak across iframes (PRD § 10.3 "atlas dies with the tab" — ADR-0003).

Three options:

| Option | Verdict | Reason |
|--------|---------|--------|
| Zustand | Adds a dependency for ~30 lines of code | Brings tree-shaking-resilient stores, but the `useSyncExternalStore` shape we use is identical without the dependency. |
| React Context + reducer | Forces every consumer of the context to re-render on every state tick (table, drawer, scope picker — all three re-render even if only the loading visualizer cares about the progress field) | Performance regression vs. fine-grained selectors. |
| **Module singleton with `useSyncExternalStore`** | **Chosen.** | Component subscribes via `useSyncExternalStore(subscribe, getSnapshot, getServerSnapshot)`. Selector pattern (per-component `getSnapshot` derives only the slice it needs) gives the same fine-grained re-render control as Zustand without the dependency. Survives strict-mode double-mount because the singleton lives at module scope. Does not leak across iframes because each iframe has its own JS heap. |

Concrete shape:

```ts
// core/atlas-store.ts (sketch — final contract recorded by Lead Developer in § 4c-1)

let state: AtlasState = { kind: 'idle' };
const listeners = new Set<() => void>();

export function subscribeAtlas(listener: () => void): () => void {
  listeners.add(listener);
  return () => listeners.delete(listener);
}

export function getAtlasSnapshot(): AtlasState {
  return state;          // referentially stable until setAtlasState is called
}

export function setAtlasState(next: AtlasState): void {
  if (next === state) return;
  state = next;
  for (const fn of listeners) fn();
}

export function resetAtlas(): void {
  setAtlasState({ kind: 'idle' });
}
```

Per-component consumer:

```ts
// in a component
import { useSyncExternalStore } from 'react';
import { subscribeAtlas, getAtlasSnapshot } from '@/core/atlas-store';

const state = useSyncExternalStore(subscribeAtlas, getAtlasSnapshot, getAtlasSnapshot);
```

The `getServerSnapshot` is the same as `getSnapshot` because all surfaces are `"use client"` and never SSR — but the third argument is required by the React API (per the Next 16 caveat in `setup/scaffold.md` Scaffold 2 step 5).

Selectors (for fine-grained re-render): components use a thin helper that derives a slice and shallow-compares.

```ts
function useAtlasSlice<T>(select: (s: AtlasState) => T): T {
  const slice = useSyncExternalStore(
    subscribeAtlas,
    () => select(getAtlasSnapshot()),
    () => select(getAtlasSnapshot()),
  );
  return slice;
}
```

The slice is recomputed by reference on every change; React's bail-out skips re-render when the result is shallow-equal.

## 8. Security Considerations

### 8.1 — Authentication and authorization

**Authentication: portal-brokered (Mode A).** No tokens in client code (NFR-5.1). The Cloud Portal injects `application.context` via the postMessage bridge; we read `resourceAccess[0].context.live` to obtain `sitecoreContextId`. There is **no Auth0 integration in this app** (per ADR-0002 + `lifecycle.md` § 5a).

**Authorization: enforced at the agent endpoints.** Per PRD § 4 "Permissions — explicit handling":
- The scan walks every page returned by `sitesGetAllPagesBySite`.
- Per-page `pagesGetComponentsOnPage` may 403 — we classify as `'forbidden'` and add to `skipped[]` with that reason. Counts are upper bounds for the editor's visible universe.
- Click-through to a page (`pages.context` mutate) is enforced by the host frame. If the editor lacks permission, Pages itself shows the appropriate error.

### 8.2 — Data handling

- **No PII in friction logs or telemetry** (NFR-5.3). Telemetry markers use page IDs and rendering IDs only. No display names. No path strings (paths could leak draft/internal page names — they are not PII per se, but we stay conservative).
- **No persistence** (ADR-0003) — there is no place for data to be exfiltrated from at rest.
- **No external network calls** outside `@sitecore-marketplace-sdk/*` (NFR-5.2). Verified at build time by inspecting `package.json` dependencies — no `fetch` calls to non-SDK origins anywhere in the source.
- **No write operations.** This app calls **only** read endpoints (`sitesGetSitesList`, `sitesGetAllPagesBySite`, `pagesGetComponentsOnPage`, `listCollections`, `retrieveSite`) plus a `pages.context` mutate that navigates the host (no content change). Per ADR-0002 forbidden list, no mutate path crosses through the scan engine.

### 8.3 — iframe / CSP

- The Marketplace portal sets the iframe's `sandbox` attribute. We check `application.context.permissions.iframe.sandbox` at startup to surface unexpected restrictions in the friction log (per `client.md` § 8c). We do not require WebGL or any sandbox capability beyond what `xmc:dashboardblocks` and `xmc:pages:context-panel` natively grant.
- We add the Chrome Local Network Access headers in `next.config.mjs` per `setup/scaffold.md` Scaffold 2 step 8.
- We do NOT combine `Access-Control-Allow-Origin: *` with `Access-Control-Allow-Credentials: true` (per scaffold step 8 — spec violation).

### 8.4 — Dependency risks

- Pin SDK version (R4 / NFR-7) — `package.json` carries exact versions of `@sitecore-marketplace-sdk/client` and `@sitecore-marketplace-sdk/xmc` after first install. Verified version recorded in `marketplace-sdk/CATALOG.md` per the catalog rule.
- Pin Blok versions implicitly via shadcn registry (each `npx shadcn@latest add` copies code into the project; subsequent SDK upgrades don't auto-update components).
- No `latest` ranges in `package.json` for SDK packages. Manual upgrade ceremony per release.

### 8.5 — Anti-metric guard at sign-off

Per DoD-4: at every release sign-off, the team explicitly compares the deployed instrumentation against the anti-metric list from PRD § 3 (no "scans/minute," "API calls served," "session count" as primary KPIs). The Team Lead (00) runs this check during `/ship`. The friction-log JSON schema (per § 5.6 telemetry pipeline) **does not** include any of those fields by construction — they're not collectable, so they can't accidentally surface on a dashboard.

### 8.6 — Cancel preserves the user's intent

`AbortController` propagates to all in-flight requests. After `abort()`:
- All `client.query(...)` calls that resolve are immediately rejected via the post-resolve `signal.aborted` check (§ 5.5/5.6).
- `Promise.allSettled` resolves with the partial results.
- `buildIndices` produces a partial atlas (`isPartial: true`).
- State transitions to `canceled`.

Per FR-5.2, the freeze is honest — what's gathered up to the cancel moment is what the editor sees. No additional requests fire after the abort.

| AR-7 | The site default-language resolver caches per-site language for the iframe lifetime; if the editor changes a site's default language mid-session, our cached value is stale until refresh | Low | Low | Acceptable. "Refresh atlas" wipes the cache. Editor mental model: refresh after structural changes. |
| AR-8 | The "branded animation" (D5) is implemented in plain CSS / SVG, not WebGL; sub-60fps on a low-spec laptop violates DoD-6 | Low | Medium | Profile early. Use `transform`/`opacity` only (GPU-composited). No layout thrash. If perf is borderline, simplify the animation rather than reach for WebGL (`client.md` § 8c — sandbox attributes may block WebGL). |
| AR-9 | The synthetic ID for unknown renderings (`unknown:<page-id>:<placeholder>:<index>`) means each "unknown" placement is a distinct row in the widget table — possibly hundreds | Medium | Medium | Group all unknown placements under a single virtual row in the widget table — `(unknown rendering)` with a count, and the drawer expands per-page. Index-build still keeps them distinct in `renderingIndex` (so totals are correct); the table-level group-by is a render concern. |
| AR-10 | The branded animation hits a sandbox restriction we didn't anticipate (e.g. `allow-modals` not granted, blocking some CSS animations on certain Chrome flags) | Low | Low | Detect sandbox at startup; fall back to a stripped-down progress bar. Surface as `'sandbox-restricted'` in telemetry. |
| AR-11 | A Site enumerated by `sitesGetSitesList` is missing from `listCollections` mapping (orphan site) — what collection does it count under? | Low | Medium | The orphan is included only when `scope.kind === 'all-collections'`; never in a collection-filtered scan. PageRef carries `collectionId?: undefined`. Document in editor-visible empty-state copy if needed. |

### 9.1 — Tradeoff: in-iframe vs. cached scans

We accept that every new tab pays the full cold scan cost (per ADR-0003). The escape hatch is Phase 3 (persistence via Standalone extension point). Rationale: the persistence layer is heavy enough that v1 should not block on it; the live-scan loading UX (§ 11 D5) is the v1 compensating mechanism.

### 9.2 — Tradeoff: shared bundle vs. per-surface bundle

We chose one Marketplace app with two surfaces (ADR-0004), which means both surfaces version-lockstep. We mitigate the bundle-bloat concern via Next.js per-route code splitting — each iframe loads only what its surface uses, plus the shared atlas core. The atlas core is ~half the code; both surfaces always pay it.

### 9.3 — Tradeoff: branded animation vs. generative-art bloom

Branded animation is simpler, more predictable, and easier to ship at 60fps. Generative-art bloom is more novel but adds rendering complexity (would lean toward `<canvas>` 2D). We pick branded animation for v1; if S6 ("felt fast") underperforms, OS-12 path opens (mini-game) — at which point the loading-visualizer interface is already designed to swap in different visualization modules.

### 9.4 — Tradeoff: module singleton vs. React Context for atlas state

Already covered in § 7.2. Tradeoff is dependency-free fine-grained reactivity vs. dropping into a state library. We chose the former.
