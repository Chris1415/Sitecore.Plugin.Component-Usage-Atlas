## 1. Design context & assumptions

- **Audience:** Content editors already using the Component Usage Atlas widget (Dashboard) and panel (Pages context panel) inside the Sitecore Cloud Portal iframe. Secondary readers of the HTML output: solution architects, IAs, external consultants — non-editors who receive a print-to-PDF artifact.
- **Goals and success criteria:**
  - **G1 (PRD-001 § 3)** — editor downloads the full widget atlas as a portable file from the widget header.
  - **G2** — editor downloads the current-page snapshot from the panel header.
  - **G4** — HTML output is human-readable for non-technical recipients without the editor having to clean it up.
  - Behavioral metrics S7 (≥30 % of completed scans yield a download), S8 (no format below 10 % or above 75 %), S10 (≥80 % of downloads complete within 1.5 s of click) gate the perceived performance of the affordances designed here.
- **Constraints (brand, a11y, stack):**
  - **Blok design system (ADR-0009).** Reuse only `@blok/*` registry primitives — no shadcn-vanilla, no Material, no hand-picked hex. Verified slugs cited in § 4.
  - **Iframe sandbox + bundle (NFR-1.4).** ≤20 KB gzipped delta total. The dropdown menu, button, popover, tooltip, and toast primitives are already in the project's `components/ui/` (verified) — incremental cost is just copy + the print stylesheet (~2-3 KB).
  - **Iframe widths (per `sitecore:blok-marketplace-integration`).** Dashboard widget surface: 300-800 px wide. Pages context panel: 320-400 px wide. Both treatments documented in § 5.
  - **WCAG 2.1 AA** (PRD-000 NFR-4 carried; PRD-001 NFR-3). Tab reach, arrow-key menu, Escape close, contrast on all states, screen-reader explanations on disabled states.
  - **No emoji codepoints** for state icons (per agent identity rule + `sitecore:blok-theming` "Color-emoji codepoints" note). The atlas already shipped two such cases in PRD-000 (`⚠`, `✓` glyphs in `widget-surface.tsx:113` and `panel-surface.tsx:368`); this spec does not add new ones — all state icons go through `@blok/icon` + `@/lib/icon` (the existing `Icon` wrapper around `@mdi/js` paths).
  - **Single proposal, divergence = subtle.** No fundamental redesign. Download button drops into the existing widget `FreshnessRibbon` right side and the panel's zone-2 action row.
- **Reference / inspiration:**
  - **Blok dropdown-menu pattern with item subtitles** — primary reference. The shipped `DropdownMenuItemTitle` + `DropdownMenuItemDescription` slots in `components/ui/dropdown-menu.tsx:101-125` already encode the exact two-line item layout the format picker needs. We reuse the same slots so the picker visually rhymes with anything else in Cloud Portal that uses them.
  - **Existing scope-filter pattern** — PRD-000 `DensityToggle` and the `Refresh atlas` button group are the visual neighbours. The Download button matches `Refresh atlas` size (`size="sm"`) and lives next to it in zone 1 / zone 2.
  - **Linear / Stripe / Vercel "export" download menus** — survey precedent: a single primary action button with an attached menu of file-format choices, item rows showing format name + one-line use-case subtitle + extension. The pattern is mature enough to require no further citation for editors who use modern SaaS daily.
  - **GitHub "Download as" menu on file blob views** — confirms the JSON / CSV / extension-suffix convention readers recognise.
- **Q&A (Team Lead pre-spawn decisions, treated as locked):**
  1. **Format-picker UX = Blok dropdown menu.** Single `Download` button → opens dropdown with three items (JSON / CSV / HTML). Each item has format name + one-line subtitle + muted `.ext` suffix. Verbatim per PRD-001 § 11.2.
  2. **Failure toast anatomy = TWO variants.**
     - **Generic failure toast** (codes `blob_construction_failed`, `unknown`, `browser_save_canceled` is benign and skipped). Sticky Blok error toast with two actions: **Retry** + **Why?** (the Why? action opens a small Blok popover with the error code's plain-English explanation and a link to the in-iframe friction log when present).
     - **Sandbox-blocked toast** (`sandbox_blocked_download` per AC-5.3, ADR-0017 F2 fallback). Sticky error toast with **THREE** actions: **Retry**, **Copy JSON to clipboard**, **Why?**. The Copy-JSON action only renders for this error code — it is the JSON-only emergency escape per ADR-0017 F2. CSV / HTML cannot be recovered through clipboard (no fidelity, no MIME); the Why? popover explains this for those formats.
  3. **Print-stylesheet typography = Blok body tokens at 11 pt.** The HTML adapter inlines `font-family: var(--font-sans), system-ui, sans-serif;` resolved to whatever `--font-sans` carries at export time (per `app/globals.css:304-308` the project resolves this to Geist via `next/font/google`). Because the print artefact is opened **outside** the editor's tab, no remote font links are emitted (AC-3.2 / NFR-4.3) — the inlined font-family stack is `system-ui, -apple-system, "Segoe UI", "Helvetica Neue", Arial, sans-serif`, which matches Blok body's resolved fallback chain when Geist is unavailable. Body size 11 pt / line-height 1.4. Heading sizes scale to `1.4rem` / `1.1rem` from the same stack. Token names referenced: `--font-sans` (Blok declared, see globals.css `@theme inline` block), body `--text-md` (= 0.875 rem = ~14 px screen, mapped to 11 pt for print), `--text-xs` (~12 px screen / ~9 pt print, used for footer + IDs).
  4. **File-size estimate in the picker = TIERED.**
     - Atlas estimated < 5 MB (the common case): NO size annotation per option. Menu stays clean.
     - 5 MB ≤ atlas < 50 MB: subtle muted text per item — `~12 MB` rendered inside the existing `DropdownMenuItemDescription` slot, suffixed after a separator dot: `Full data, machine-readable · ~12 MB`. Token: `text-subtle-text` (= `var(--muted-foreground)` per globals.css:185).
     - Atlas ≥ 50 MB: a Blok warning `@blok/icon` (`name="alert-triangle"` — exposed via `@/lib/icon` wrapper using `mdiAlert` from `@mdi/js`, matching how `widget-surface.tsx:154` already renders icons through `<Icon path={mdiRefresh} size={0.75} />`) inline-prefixed to the description, with copy `Large — may take a moment · ~78 MB`. Color token: `text-warning-fg` (= `var(--warning-foreground)`, globals.css:523) — same colour the existing skipped-link warning glyph uses on widget-surface.tsx:178.
- **Assumptions made (called out so they can be challenged):**
  - **A-UI-1:** The atlas's in-memory size estimate is computable cheaply at menu-open time as `JSON.stringify(atlas).length / 1.4` (the 1.4 factor is the empirical CSV/HTML overhead vs. JSON). This is a heuristic, not exact — acceptable for the picker because the user only needs an order-of-magnitude warning. If the heuristic costs more than 10 ms on a 50k-page atlas, fall back to estimating from `atlas.totals.pages × 8 KB` per page. Hand off to lead developer for empirical pick.
  - **A-UI-2:** Menu width: 320 px on widget surface, 280 px on panel surface. Driven by the longest expected description line at the 50-MB warning tier (`Lite data, spreadsheet-friendly · ~78 MB` ≈ 38 char). Panel is intentionally narrower because the panel iframe is itself ≤400 px and the menu must not overflow the iframe edge.
  - **A-UI-3:** "Why?" popover uses `@blok/popover` (already installed at `components/ui/popover.tsx`) anchored to the Why? action button inside the toast, with `align="end"` and 240 px width. Closes on outside-click or Escape.
  - **A-UI-4:** The download button enabled-during-refresh-with-prior-atlas behaviour (AC-1.1 / AC-2.1) is signalled by **no badge, no spinner** — the button looks fully enabled and clicking it produces the previous atlas's snapshot. Rationale: PRD-001 explicitly says the prior atlas is a valid export target during refresh, so we don't communicate "stale" visually — the JSON `scan_timestamp` already reveals the source-of-truth time to anyone who cares.
  - **A-UI-5:** The `Why?` action label is the literal string `Why?` (with the question mark) — short enough for the toast's tight horizontal budget; matches how Linear, GitHub, and Stripe use trailing-question-mark micro-copy for explanatory affordances.
  - **A-UI-6:** Toast linger time: 4 s for success, sticky-until-dismissed for failure. Matches PRD-001 § 11.4 verbatim — flagged below as OQ-6 still nominally open (user input not collected) but spec commits to 4 s.

## 2. Design direction

### Visual direction — **"Quiet Affordance"**

The export feature is **additive**, not headlining. Editors are mid-task when they need it; the affordance must be discoverable but not visually competing with the atlas's existing scan-status / freshness-ribbon / search input. The named direction is **Quiet Affordance**: a single Blok-secondary download button that visually rhymes with `Refresh atlas`, opens a dense informative menu, and disappears back into the chrome on completion.

Rationale: PRD-000 already established the widget's "freshness ribbon → table" hierarchy (widget-surface.tsx:91-160) and the panel's "page-context card → impact list" hierarchy (panel-surface.tsx:347-419). A loud export button would compete with those primary signals. A quiet one inherits authority from neighbours.

### Layout and structure

| Surface | Where the new affordance sits | Vertical position |
|---|---|---|
| **Widget** | Right side of the existing `FreshnessRibbon` row (widget-surface.tsx:146-157), to the LEFT of `Refresh atlas`. Order: `[Download button] [Refresh atlas button]`. Reasoning: download is read-only / data-out; refresh is mutation-y / data-in; convention is "view actions left of mutation actions". | Zone 1 (top of surface) |
| **Panel** | Right side of zone 2's existing flex row (panel-surface.tsx:385-398), to the LEFT of the skipped-link warning if present, sharing space with `Refresh` (which the panel surfaces via the page-context-card's `onRefresh`). Order in zone 2: `[Direct-bindings affordance] ··· [Download icon-only] [Skipped warning if present]`. | Zone 2 (just under freshness, above the rendering tree) |

No structural changes to either surface beyond inserting the button (and its dropdown portal) into the existing flex containers.

### Key components and patterns

- **`@blok/button`** — `variant="outline" colorScheme="neutral" size="sm"` on widget; `variant="ghost" colorScheme="neutral" size="icon-sm"` on panel (icon-only). Citation: `components/ui/button.tsx:33-160` defines all four variants and three sizes.
- **`@blok/dropdown-menu`** — composes the format picker. Uses `DropdownMenu` + `DropdownMenuTrigger asChild` + `DropdownMenuContent` + three `DropdownMenuItem`s, each containing `<Icon>` + `DropdownMenuItemText` (with `DropdownMenuItemTitle` + `DropdownMenuItemDescription`). Citation: `components/ui/dropdown-menu.tsx:88-125` (these slots already exist).
- **`@blok/icon`** (resolved through the project's `@/lib/icon` wrapper around `@mdi/js`) — download icon = `mdiDownload`; warning glyph = `mdiAlert`; per-format glyph (optional, see § 4) = `mdiCodeBraces` (JSON), `mdiTable` (CSV), `mdiFileDocumentOutline` (HTML).
- **`@blok/tooltip`** — wraps the icon-only panel button and the disabled-state buttons on both surfaces. Already installed at `components/ui/tooltip.tsx`.
- **`@blok/popover`** — backs the **Why?** action's explanation surface. Already installed at `components/ui/popover.tsx`.
- **`@blok/sonner`** (or equivalent toast primitive registered as `@blok/toast`) — backs success and error toasts. **NOT YET INSTALLED** in the project (verified — `components/ui/{toast,sonner,toaster}*` glob returned 0 hits, but `globals.css:303` already declares `--color-sonner-error`, indicating Blok's intent for this slot). The lead developer must add `@blok/sonner` (kebab-case slug per `sitecore:blok-components` "Overlays & dialogs" list) before implementation. The print stylesheet does not depend on this; the in-iframe success/failure UX does.
- **`@blok/skeleton`** is **not** used here — construction is short enough (<1.5 s, NFR-1.1) that a button-spinner is sufficient; a skeleton would oversell the wait.

### Critical user flows

**Flow A — Happy path, widget, JSON:**
1. Editor completes a scan → atlas in-memory.
2. Editor moves cursor to widget header, sees `Download` button (S1).
3. Editor clicks → menu opens (S2). Three options visible. Atlas size 2.1 MB → no size annotation per the < 5 MB tier.
4. Editor presses Down arrow twice, lands on `CSV`, presses Enter.
5. Menu closes. Button enters spinner state (S3). Construction takes ~400 ms.
6. Browser save dialog appears OR file lands in Downloads folder (per AC-1.3, ADR-0017 primary mechanism).
7. Toast S6 surfaces: "Downloaded `atlas-acme-widget-all-collections-20260503T101441Z.csv`." Auto-dismisses in 4 s.

**Flow B — Sandbox-blocked path, panel, JSON:**
1. Editor on a Pages canvas with the panel open.
2. Clicks the icon-only Download button (S4). Tooltip on hover read "Download snapshot".
3. Menu opens (S5). Editor picks `JSON`.
4. Construction succeeds, but the synthetic `<a>` click is silently rejected (ADR-0017 detection contract — 5-s heuristic fires, telemetry logs `export.fail` with `errorCode: 'sandbox_blocked_download'`).
5. Sandbox-blocked toast S8 surfaces with three actions.
6. Editor clicks `Copy JSON to clipboard`. The same Blob body (already constructed) is `navigator.clipboard.writeText(...)`'d. Toast updates copy in-place to "Copied JSON to clipboard." and the action set collapses to `[Why?]` only. Auto-dismisses in 4 s.
7. If editor instead clicks `Why?`: popover S9 anchors to the action button, explains the sandbox restriction in 2 sentences, links to the friction log when ADR-0013 ring buffer has anything.

**Flow C — Empty atlas:**
1. Editor scanned a tenant with zero renderings (rare but valid per IS-18).
2. Download button still enabled.
3. Menu opens, no size annotation, all three format options present.
4. Editor picks HTML.
5. Construction emits the IS-18 contract: header + `(No renderings found.)` body line.
6. Toast S6 surfaces: "Downloaded `atlas-acme-widget-all-collections-20260503T101441Z.html` — empty atlas."

**Flow D — Refresh-in-progress:**
1. Atlas previously completed; editor clicked `Refresh atlas`; new scan running.
2. Download button stays enabled with no spinner (per A-UI-4).
3. Editor clicks → menu opens → picks JSON.
4. Construction reads the **prior** atlas at click time (per FR-1.2 / AC-1.1). Snapshot reflects pre-refresh state with `is_partial: false` and the previous `scan_timestamp`.
5. Toast S6 surfaces normally; refresh continues uninterrupted in the background.

**Flow E — Mid-scan navigation on panel:**
1. Editor on panel, page X open, clicks Download → JSON.
2. Mid-construction, editor clicks a different page → panel `pages.context` subscription updates `activePageId`.
3. Construction continues against the click-time `surfaceContext` clone (FR-5.3 / AC-2.7), emits the snapshot for page X.
4. Toast S6 surfaces page-X filename. Panel re-renders for the new page in parallel (no flicker on the toast or on the rendering tree because they live in independent React state).

### Responsiveness

| Breakpoint | Widget surface | Panel surface |
|---|---|---|
| **300 - 480 px (narrow widget / panel)** | Button collapses to icon-only (`size="icon-sm"`), label hidden, tooltip surfaces it. Menu width drops to 280 px. | Always icon-only by default. Menu width 280 px. |
| **480 - 800 px (typical widget)** | Button shows label `Download` next to the icon. Menu width 320 px. | n/a |

## 4. Component and pattern spec

### 4.1 — `<DownloadButton>` (new)

Renders the trigger. Two variants, one for each surface.

| Prop | Type | Notes |
|---|---|---|
| `surface` | `'widget' \| 'panel'` | Drives variant + size selection. |
| `state` | `'enabled' \| 'disabled-no-data' \| 'disabled-panel-loading' \| 'disabled-scan-in-progress-no-prior' \| 'constructing'` | See state table below. |
| `atlasSizeBytes` | `number \| null` | Used by the menu to compute the tiered size annotation. `null` skips. |
| `onSelectFormat` | `(format: 'json' \| 'csv' \| 'html') => void` | Caller wires the construction trigger. |

**Composition (widget):**

```
<DropdownMenu>
  <DropdownMenuTrigger asChild>
    <Button variant="outline" colorScheme="neutral" size="sm" disabled={state !== 'enabled'} aria-label="Download snapshot">
      {state === 'constructing' ? <Spinner /> : <Icon path={mdiDownload} size={0.75} />}
      <span>Download</span>
    </Button>
  </DropdownMenuTrigger>
  <DropdownMenuContent align="end" className="w-80">
    {/* three items — see § 4.2 */}
  </DropdownMenuContent>
</DropdownMenu>
```

**Composition (panel):**

```
<DropdownMenu>
  <DropdownMenuTrigger asChild>
    <Tooltip content={tooltipForState(state)}>
      <Button variant="ghost" colorScheme="neutral" size="icon-sm" disabled={state !== 'enabled'} aria-label="Download snapshot">
        {state === 'constructing' ? <Spinner /> : <Icon path={mdiDownload} size={0.75} />}
      </Button>
    </Tooltip>
  </DropdownMenuTrigger>
  <DropdownMenuContent align="end" className="w-72">
    {/* three items — same content as widget, narrower wrap */}
  </DropdownMenuContent>
</DropdownMenu>
```

#### State table

| State | Visual | Tooltip / aria copy | Trigger |
|---|---|---|---|
| `enabled` | Default Blok outline button (widget) / ghost icon-button (panel). Icon `mdiDownload`. | `Download snapshot` | Atlas in memory (or refresh-with-prior). |
| `disabled-no-data` | `disabled` attribute, opacity 0.5 (Blok default per button.tsx:25). Icon `mdiDownload`. | `No data — start a scan first.` | `state.kind === 'idle'` AND no prior atlas. Per AC-1.1 / AC-2.1. |
| `disabled-scan-in-progress-no-prior` | Same disabled visual. | `Scan in progress — finish the scan to download.` | `state.kind === 'scanning'` AND no prior atlas. PRD-001 § 11.3. |
| `disabled-panel-loading` (panel only) | Same disabled visual. | `Loading current page…` | Panel: `activePageId` resolved but per-page fetch in flight (panel-surface.tsx:305-307). Per AC-2.1. |
| `constructing` | Disabled, icon swapped for inline spinner (Blok motion: `animate-spin` over the existing icon size, `text-muted-foreground`). | `Preparing your download…` | Set when `onSelectFormat` is called, cleared on toast emission (success or fail). |

aria-describedby points to a visually-hidden `<span>` carrying the same copy as the tooltip on every disabled state, per NFR-3.3 / 11.6.

### 4.2 — Format-picker menu items

Three items in fixed order: **JSON**, **CSV**, **HTML**. Each item composed as:

```
<DropdownMenuItem onSelect={() => onSelectFormat('json')}>
  <Icon path={mdiCodeBraces} size={0.7} aria-hidden />
  <DropdownMenuItemText>
    <DropdownMenuItemTitle>JSON <span className="text-subtle-text font-normal">.json</span></DropdownMenuItemTitle>
    <DropdownMenuItemDescription>Full data, machine-readable{sizeAnnotation}</DropdownMenuItemDescription>
  </DropdownMenuItemText>
</DropdownMenuItem>
```

`sizeAnnotation` is computed at menu-render time per the tier rule:

| Tier | `sizeAnnotation` rendered text | Decoration |
|---|---|---|
| `< 5 MB` | empty string | none |
| `5 MB ≤ s < 50 MB` | ` · ~12 MB` (replace 12 with computed) | none — relies on existing `text-subtle-text` for muted feel |
| `≥ 50 MB` | ` · ~78 MB — Large, may take a moment` | warning icon prefixed inside the description: `<Icon path={mdiAlert} size={0.55} className="text-warning-fg mr-1" aria-hidden />`. Color token `text-warning-fg` (= `var(--warning-foreground)`, globals.css:523). Same colour shipped on widget-surface.tsx:178. |

Per-format details:

| Format | Title | Subtitle copy | Per-item icon | Filename suffix |
|---|---|---|---|---|
| JSON | `JSON .json` | `Full data, machine-readable` | `mdiCodeBraces` | `.json` |
| CSV | `CSV .csv` | `Lite data, spreadsheet-friendly` | `mdiTable` | `.csv` |
| HTML | `HTML .html` | `Lite data, printable / shareable` | `mdiFileDocumentOutline` | `.html` |

PRD-001 § 11.2 subtitle copy reused verbatim. The `.ext` suffix lives inside the title slot rendered with `text-subtle-text font-normal` so the format name stays bold-ish (Blok title slot is `font-semibold` per dropdown-menu.tsx:108) and the extension reads as muted ornament.

Menu interaction (verified against `components/ui/dropdown-menu.tsx` — Radix-backed):

- **Tab** to button.
- **Enter** / **Space** opens the menu; first item auto-focused.
- **Arrow Down/Up** moves focus between items (Radix default).
- **Enter** / **Space** on focused item invokes `onSelect` and closes the menu.
- **Escape** closes the menu, restores focus to the trigger.
- **Click outside** closes the menu.

Per NFR-3.2: the visually rendered title on each item already encodes both the format name and a use-case clause, so screen readers announce `JSON, .json, Full data, machine-readable` (or browser equivalent) without aria-label workarounds.

### 4.5 — Sandbox-blocked toast (S8)

Same shell as S7 with one extra action and a tweaked body:

| Property | Difference from S7 |
|---|---|
| Body copy | `Downloads are blocked in this app's iframe. Copy the JSON to clipboard, or contact Sitecore support.` |
| Actions (in order) | **Retry** + **Copy JSON to clipboard** + **Why?** |
| Copy-JSON action visibility | ONLY when `errorCode === 'sandbox_blocked_download'`. ADR-0017 F2 fallback. CSV / HTML failures of this kind do not show Copy-JSON because clipboard cannot carry their formats with fidelity — the Why? popover explains why for those formats. |

The Copy-JSON action handler:
1. Call the JSON adapter (already pure per FR-1.1) against the same `surfaceContext` the failed click captured.
2. `navigator.clipboard.writeText(jsonString)`.
3. On success: replace toast body with `Copied JSON to clipboard.`, drop Retry + Copy actions, keep Why? for 4 s, then auto-dismiss.
4. On clipboard failure (rare; permission denied): swap to a tertiary toast `Clipboard blocked. Open the friction log for the raw JSON.` — terminal state, sticky. Telemetry: `export.fail` with `errorCode: 'clipboard_blocked'` (new union member; lead developer to add).

#### 4.7.7 — Long-list relationship-visibility check (per agent identity rule)

The HTML output's `<table>` may render hundreds of rendering rows. The widget output is a flat list (one row per rendering, columns are aggregates). The panel output is also flat (one row per rendering on the current page) but each row implicitly relates to a datasource (column 3-4: `datasource_id`, `datasource_display_name`). At 200+ rows, can the recipient still trace which datasource binds which rendering?

**Answer: yes, because the relationship is per-row, not cross-list.** The table is wide enough that each row is self-contained: rendering ID + display name + datasource ID + datasource display name + counters live on the same row. There is no parallel "list of renderings" + "list of datasources" pattern in the print artefact (that pattern only exists in the live panel zone-3, where PRD-000 already adopted nested expansion per the prior live-tenant feedback). The flat-table choice for HTML is the right one for print readability — a recipient reading row 217 still sees the rendering↔datasource binding in the same line.

For the JSON output, the relationship is structurally explicit (`renderings[].datasources[]` for widget; `renderings[].bound_datasource` for panel) — also unambiguous.

### 4.10 — Accessibility notes (component-level)

- **`<DownloadButton>`** — `aria-label="Download snapshot"` always present (icon-only variant relies on it; labelled variant carries it as redundancy for screen readers that prefer aria-label over the visible text). `aria-haspopup="menu"` (Radix dropdown sets this automatically). `aria-expanded` toggles with menu open state.
- **Disabled-state tooltips** — `aria-describedby` points to a visually-hidden `<span>` with the same copy, per NFR-3.3. The tooltip alone is not sufficient because some screen readers do not announce `title`/`@blok/tooltip` content reliably on disabled controls.
- **Format picker** — Radix dropdown handles ARIA roles (`role="menu"`, items `role="menuitem"`). Each item's accessible name is the title text + description text concatenated naturally; we do not add aria-label overrides because the visible content already conveys the choice fully.
- **Toasts** — `aria-live="polite"` for success, `aria-live="assertive"` for errors. Action buttons inside toasts are keyboard-reachable (Tab from elsewhere; some toast libraries auto-focus the first action — verify in implementation).
- **"Why?" popover** — focus trap inside popover while open; first focusable element is the body's optional friction-log link if present, else the close affordance. Escape closes and restores focus to the Why? action button.

### 5.1 — Breakpoints
