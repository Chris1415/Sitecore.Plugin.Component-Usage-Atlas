> **Reading note.** This is one of three divergent variants. v3 takes the most opinionated stance on tone: the atlas speaks to the editor in plain English first and shows tables only when needed. It honors every architectural lock (ADR-0002..0014) and only diverges from v1/v2 in **composition, layout, motion, typography rhythm, and copy voice** — never in palette, fonts, or component library. All UI uses real `@blok/*` primitives and real Blok semantic tokens.

## 1. Design context & assumptions

### Audience

Primary: **Content Editors** in Sitecore Pages (per PRD § 4). They are not technical implementers, they live inside Pages most of the working day, and they think in *pages*, not *items*. The "Editor's Whisper" direction targets this audience explicitly: the app reads to them like a colleague's annotated note rather than a dashboard report. A-PERS-2 ("editors recognize rendering names by display name") is leaned into here — display names are the headline; IDs are tooltips.

Secondary (incidentally served, not optimized for): information architects doing component rationalization and developers preparing migrations. The conversational layout still works for them — they will skim faster, but the same primitives surface every datapoint.

### Goals and success criteria

Tied to PRD §§ 3 and 8:

- **G1 (component-centric answer)** — the widget must lead with a search; the answer per rendering is a card with a sentence summary, expandable to a per-page list.
- **G2 (page-centric answer)** — the panel must read top-down as a paragraph + list, with cross-tenant counters that look like inline pills inside sentences.
- **G3 (datasource impact pre-change)** — datasource impact is its own annotated section in the panel, never folded into the rendering list.
- **G4 (bearable wait)** — the loading visual is a single line of running copy ("Reading sites… Now mapping pages… Looking at components on page 47 of 312…"). The wait is *narrated*, not decorated.
- **S6 ("felt fast")** — narration drives perceived speed: progress text changes ≤ every 1.5s (matches AC-4.2). 60fps via CSS transitions only.
- **NFR-4 (a11y AA)** — every interactive element keyboard-reachable; `aria-live="polite"` on progress narration; no color-only signals; the conversational tone does not cost screen-reader users any clarity (the same content is in the DOM, in reading order).

### Constraints (brand, a11y, stack)

- **Stack** — Next.js 16 App Router (per ADR-0008); Blok `@blok/*` registry components only (per ADR-0009); module-singleton pub/sub (per ADR-0010); branded animation only — no generative bloom, no mini-game (per ADR-0011); two surfaces, one app (per ADR-0004); no backend, no persistence (per ADR-0002 / ADR-0003); rendering = definition ID, datasource = item ID (per ADR-0005); direct bindings only with a mandatory `ⓘ` affordance (per ADR-0006).
- **Design system** — Blok semantic tokens (`bg-background`, `bg-card`, `text-foreground`, `text-muted-foreground`, `bg-primary`, `text-primary`, `bg-muted`, `border-border`, `ring-ring`, `bg-destructive`, `bg-warning` if exposed by Nova preset — fall back to `bg-primary` + iconography otherwise). No hand-picked hex anywhere. No font families beyond what `theme.json` ships.
- **Iframe constraints** — both surfaces render inside a Cloud Portal iframe; widths to test: **800px, 1024px, 1440px** (per Blok `theming.md` Marketplace section). Shadows kept ≤ `shadow-sm` on app-shell surfaces.
- **A11y target** — WCAG 2.2 AA. Keyboard nav, focus-visible rings via Blok's `ring-ring`, `aria-live` on progress, `aria-expanded` on result cards, `aria-pressed` on toggles, no color-only state.
- **Light + dark parity** — Blok `theme.json` flips automatically under `.dark`; the variant adds zero dark-mode work beyond honoring tokens.

### Reference / inspiration

Adapted, never copied:

- **Notion** — conversational empty-state copy and inline pill counters inside paragraphs ("This database has **12** entries"). Notion's calm, sentence-first interface inside a workspace tool.
- **Stripe Dashboard** — the calmer pages (Customer detail, Payment timeline) use small cards ordered chronologically with quiet labels and inline numbers, not stat tiles.
- **Linear** — narrow, ink-on-paper panels with subtle hover affordances; status as inline badges, not columns.
- **Editorial longform on the web** (NYT Magazine, MIT Technology Review feature articles) — generous line-height, large display headlines, sparse color, eyes pulled by typography rhythm rather than hue.
- **macOS Mail "Smart Mailbox" empty states** — friendly one-line headers and one-button affordances; the screen feels finished even when sparse.

Adapt (not copy): the rhythm is the borrowed thing — display heading → calm body sentence → discrete card list. Colors and primitives stay 100% Blok.

### Answers to clarifying questions

The following were established in the variant assignment and confirmed against the PRD/architecture; they are not user-facing questions.

- **Q: How tonal can the copy be?** — A: Opinionated and friendly, but never cute. "What component are you looking for?" yes; "Hey, friend!" no.
- **Q: Cards or rows for widget results?** — A: Cards (one per rendering). Each card is a small editorial unit with a sentence summary; expanding shows the full per-page list. This is the deliberate v3 divergence vs. v1/v2 (likely table-based).
- **Q: Counters as stat blocks or as inline pills?** — A: Inline pills inside sentences. "Used on **12** pages across **3** sites." The pill carries the count; the sentence carries the meaning.
- **Q: Where does scope picker go on the widget?** — A: Below the search input, smaller and quieter. The scope picker is a refinement, not a primary entry point.
- **Q: How does the panel read?** — A: Top-summary paragraph → renderings list (cards with pills) → datasource impact section with its own annotated header. The panel reads top-to-bottom like an annotated margin.
- **Q: Loading visual?** — A: Single line of running narration with a tiny progress underline beneath it; cancel is a quiet "Stop" link, not a button.

### Assumptions made (label-friendly so QA can challenge)

- **A-V3-1** — Editors prefer reading "Used on **12** pages across **3** sites" over scanning a row with three numeric columns. Validation: qualitative pulse post-launch; if S6 dips on this variant in a head-to-head test, fall back to denser rows. (Tied to A-PERS-2.)
- **A-V3-2** — Card-per-rendering is acceptable at scale: tenants with hundreds of distinct renderings will be navigated via search, not by scrolling a card list. The card list is virtualized (or windowed) so 1k cards are not a perf hit. Validation: NFR-1.2 still met (≤50ms filter response).
- **A-V3-3** — Sentence narration on the loading screen reads as more reassuring than a phase-ring or progress bar. Validation: S6 ("felt fast") on this variant; A/B against v1/v2 if both are clickable POCs.
- **A-V3-4** — A "Stop" link reads as cooperative; a "Cancel" button reads as adversarial. Editors are more likely to wait through to completion when the affordance to bail feels gentle. Validation: S4 (intentional-cancel split) on this variant.
- **A-V3-5** — The narration text changing every ~750ms (well under AC-4.2's 1.5s ceiling) is enough to make a 30s wait feel like "the app is doing things." Validation: S6.
- **A-V3-6** — Inline pills carry enough visual weight without color saturation. The pill background is `bg-muted` by default; only the *active* / hovered pill flips to `bg-primary` `text-primary-foreground`. Validation: WCAG AA contrast on muted surface (verified in QA).

## 2. Design direction

### Visual direction — "Editor's Whisper"

A calm, conversational, typographic dashboard. The atlas reads to the editor in plain sentences before it shows them tables. Generous whitespace, large display headings (Blok's display weight), and a clear typographic rhythm pull the eye top-down. Counters appear as **inline pills inside sentences** — `Used on **12** pages across **3** sites.` — not as stat tiles. Color is sparse: most of the surface is `bg-background` + `text-foreground`; `bg-primary` is reserved for hover/focus states and the loading underline. The app has a voice — friendly, slightly opinionated, never cute.

The variant's name comes from how the panel reads: like a margin note in a careful reader's book. Short paragraphs, then a list, then a quieter aside. The widget reads more like the cover page of an article — a question ("What component are you looking for?"), a search field, a calm grid of result cards.

### Layout and structure

#### Widget — `xmc:dashboardblocks` (route `/widget`)

Centered hero with breathing room. Top-down rhythm:

1. **Greeting line** — `text-foreground`, Blok display heading: *"What component are you looking for?"* Reads as a typographic asset, not a label.
2. **Search input** — full-width, large (`@blok/input` with size `lg` if exposed; otherwise `@blok/search-input`), placeholder *"Search by name, e.g. HeroV2 or Promo Banner."* Centered, ~640px max-width on wide viewports.
3. **Scope row** — quieter, smaller font (`text-muted-foreground` `text-sm`). Reads: *"Across all collections."* with the collection name as an inline `@blok/select` (ghost variant) acting as an inline link. A second sentence carries the freshness: *"Last scanned at 14:32. Refresh atlas."* — "Refresh atlas" is a `@blok/button` variant `ghost` with `text-primary` on hover.
4. **Result rail** — a vertical list of result cards (`@blok/card`), one per rendering. Each card:
   - Headline: rendering display name (Blok `text-lg`, `font-semibold`). Disambiguation suffix `· abc1234` per FR-9.2 appears as a `@blok/badge` (variant `bold`, `colorScheme="muted"`) inline-right.
   - Subhead sentence: *"Used on **12** pages across **3** sites. Bound to **2** datasources."* Counters are `@blok/badge` variant `default`, `colorScheme="primary"` rendered inline as small pills.
   - Expand affordance: a single quiet *"Show pages"* link aligned right (`text-primary text-sm`, `underline-offset-4` on hover).
   - Expanded state: the card grows downward, revealing the page list (one row per page, `@blok/separator` between rows) split into two collapsible groups — *"Pages that place this rendering directly"* (uses `@blok/collapsible`) and *"Pages that also bind one of its datasources"* (also `@blok/collapsible`).
5. **Footer affordance** — at the bottom of the rail: *"Showing **47** of 47 renderings."* with the inline pill counter. Below that, the v1 affordance per ADR-0006: *"ⓘ Direct bindings only."* — clickable, opens a `@blok/tooltip` with the FR-8.3 explanatory copy.

No table component on the widget. The cards *are* the data display.

#### Page Context Panel — `xmc:pages:context-panel` (route `/panel`)

Narrow column (~360px typical iframe width). Reads top-down like an annotated margin. Top-down rhythm:

1. **Top-summary paragraph** — Blok `text-base`, leading-relaxed. Two sentences:
   *"This page renders **5** components. **Two** of them are widely shared; **three** are unique to this page."*
   The numbers are inline `@blok/badge` pills.
2. **Freshness sentence** — `text-sm` `text-muted-foreground`: *"Last scanned 14:32. Counters update once the scan finishes."* If the scan is still running, the cross-tenant counters render as `@blok/skeleton` rectangles inside the card pills.
3. **Renderings list** — small card per rendering (`@blok/card` with reduced padding via `p-3`). Each card:
   - Headline: rendering display name (`font-medium`, `text-sm`).
   - Sentence: *"Also on **12** other pages."* (counter pill inline). If the count is 0, the sentence is *"Unique to this page."*
   - Datasource sub-line (only if a datasource is bound directly): *"Datasource: **Heroes/Spring**"* — datasource name as a `@blok/badge` ghost variant, link-styled, opens the datasource detail in the drawer.
   - Parameter tooltip (per AC-2.1): a small `?` `@blok/tooltip` icon to the right of the rendering name; hover/focus shows the parameter map.
4. **Quiet separator** — `@blok/separator` with `bg-border` plus a small `text-xs text-muted-foreground` annotation: *"Datasource impact"*.
5. **Datasource impact section** — same card pattern, but each card opens with a sentence about the datasource, not a rendering: *"`Heroes/Spring` is bound on **5** pages across **2** sites — including this one."* If `isMissing: true`, the card's left edge gains a `border-l-2 border-warning` (or `border-destructive` if warning token is unavailable in the Nova preset) and the sentence prefix becomes *"⚠ Missing datasource — referenced by **5** pages."* per AC-2.5. The warning is *also* signaled by an icon and the prefix word — never color alone (NFR-4.3).
6. **Foot affordance** — *"ⓘ Direct bindings only."* — same pattern as the widget.

#### Loading view — both surfaces

A single line of running copy at the visual center of the surface, surrounded by extreme whitespace:

```
Reading sites…           ← changes ~every 750ms
─────────────             ← thin progress underline (2px, bg-primary), animates left-to-right
Stop                      ← quiet text link, text-muted-foreground hover:text-primary
```

Narration script (cycles based on phase + counters from `core/scan-engine.ts`):

- Phase `sites`: *"Reading sites…"* → *"Found **N** sites."*
- Phase `pages`: *"Mapping pages in **siteName**…"* → *"Listed **M** pages so far."*
- Phase `components`: *"Looking at components on page **i** of **N**."* (updated every ≥200ms per `ScanProgress.elapsedMs`) → *"Almost there — **k** pages left."*

The underline beneath the line is a single `<div>` with `width: ${progress}%; transition: width 200ms linear;` driven by `pagesCompleted / pagesTotal`. The underline color is `bg-primary`. CSS-only transition — no canvas, no requestAnimationFrame loop — directly satisfies the ADR-0011 *"push to CSS animations or a low-cost requestAnimationFrame loop"* and the 60fps target via DoD-6.

Stall hint (per AC-4.4): if no progress for >5s, the line softly fades to *"Slow connection — keeping at it. **Try again** if you'd rather restart."* — *Try again* is a `@blok/button` variant `ghost` size `sm`, inline.

Cancel-state result page: *"Stopped at **N** of **M** pages. **Show what we have** or **Re-scan**."* (per US-1 AC-1.2 and PRD § 11.3 *Scan canceled*). Both affordances are `@blok/button`, *Re-scan* is `variant=primary`, *Show what we have* is `variant=ghost`.

### Key components and patterns

- **Conversational hero** — display heading + search input + quiet scope + freshness line. Replaces a "page title + filter row" pattern.
- **Sentence cards** — `@blok/card` with sentence-first content; counters as inline `@blok/badge` pills inside the sentence. Expands via `@blok/collapsible` to reveal the page list.
- **Inline pill counter** — `@blok/badge` (variant `default`, `colorScheme="primary"`) used inside running text. The pill is the sole numeric affordance — there are no stat blocks anywhere in v3.
- **Quiet refresh / scope** — link-styled controls, not buttons. They sit in the freshness sentence ("Last scanned 14:32. **Refresh atlas**.") rather than as a toolbar.
- **Drawer (`@blok/sheet`)** — same architectural pattern as the rest of the app, but the drawer's *header* is a sentence too: *"**HeroV2** appears on **12** pages."* The body splits into the two `@blok/collapsible` groups (direct vs. via-datasource) per AC-1.6.
- **Branded loading line** — narration line + thin progress underline + Stop link. CSS-only animation. The "Distract me" toggle is absent in v1 per ADR-0011.
- **Information affordance (`ⓘ Direct bindings only`)** — small inline icon-link rendered with `@blok/tooltip`, copy from FR-8.3. Mandatory on both surfaces wherever datasource counts appear.

## 3. Key screens and flows

### Screens

| Screen | Surface | Purpose |
|---|---|---|
| `/widget` — Greeting / empty atlas | Widget | First mount on a fresh tab. Displays the greeting headline + search + scope; behind it the loading view runs. |
| `/widget` — Loaded result rail | Widget | Scan complete; cards listed with sentence summaries; search filters in place. |
| `/widget` — Card expanded | Widget | One card opened; per-page list visible inside the card; drawer optional for full detail. |
| `/widget` — Drawer (rendering detail) | Widget | Opened via card "Show pages"; shared with panel via `<UsageDrawer>`. |
| `/widget` — Empty search | Widget | Search yields zero matches; empty-state line per PRD § 11.3. |
| `/widget` — Cancel partial | Widget | Cancel-state result page; partial atlas surfaced with prompts to act or re-scan. |
| `/widget` — Error | Widget | "Atlas needs a tenant connection." sentence + retry link (NFR-3.2). |
| `/panel` — Loading | Panel | Same loading line as widget; counters appear as skeletons in renderings list. |
| `/panel` — Loaded | Panel | Top summary paragraph + renderings list + datasource impact section. |
| `/panel` — Drawer (counter click) | Panel | Same `<UsageDrawer>` as widget. |

### Critical user flows

1. **Pre-publish impact check on a page (US-2 / G2 / G3).**
   Editor opens Pages → context panel mounts → reads top-summary sentence → spots that `HeroV2` is "Also on **12** other pages" → clicks the pill → drawer opens → editor sees the 12 pages → decides whether to publish.

2. **Pre-deletion lookup (US-1 / G1).**
   Editor opens dashboard → widget greets *"What component are you looking for?"* → editor types `LegacyPromo` → result card appears: *"Used on **0** pages."* → editor confirms safe to delete.

3. **Datasource change impact (US-2 + AC-2.4 / G3).**
   Editor on a page → opens panel → scrolls past renderings to "Datasource impact" → sees `Heroes/Spring is bound on **5** pages across **2** sites — including this one.` → clicks counter pill → drawer lists the 5 pages → opens an out-of-current-page page in Pages.

4. **Scope narrowing (US-3).**
   Editor opens widget → sees freshness sentence including *"Across all collections."* → clicks the inline collection name → `@blok/select` ghost-variant dropdown drops with collection options → selects one → loading line resumes ("Reading sites in **Marketing collection**…") → result rail re-renders against the narrower scan.

5. **Cancel a slow scan (US-4 / S4 — intentional cancel).**
   Editor watches narration line → loses patience → clicks "Stop" link → cancel-state result page shows *"Stopped at **47** of **312** pages. **Show what we have** or **Re-scan**."* → editor clicks *Show what we have* → partial result rail renders with an `@blok/badge` "Partial" suffix on each card.

6. **Slow-connection stall (US-4 / AC-4.4).**
   No progress for 5s → line softly fades to *"Slow connection — keeping at it. **Try again** if you'd rather restart."* → if editor ignores, scan continues; if editor clicks *Try again*, scan aborts and re-runs.

7. **Forbidden / skipped pages (US-1 AC-1.5 + AC-4.5 / IS-17).**
   Scan completes with `skipped[]` non-empty → freshness sentence appends: *"**3** pages skipped — see why."* The "see why" link opens a sub-drawer listing skipped pages and reasons (`forbidden | timeout | not_found | network_error | other`) — text labels, never color alone.

### Responsiveness

| Width | Widget | Panel |
|---|---|---|
| **800px** (narrow iframe) | Greeting headline scales to `text-3xl`. Search input full-width. Cards full-width with reduced padding. Pills wrap to a second line if the sentence overflows. | Renderings cards condense — datasource sub-line wraps. Drawer becomes near-full-width (per Blok's `Sheet` mobile behavior). |
| **1024px** (typical Cloud Portal iframe) | Headline `text-4xl`. Search input ~640px max-width centered. Result rail full-width. Card body has comfortable padding. | Two-column rendering meta-row inside the card (rendering name on left, counter pill aligned right). |
| **1440px+** (widescreen) | Headline `text-5xl`. Generous side-margins on the result rail (`max-w-3xl mx-auto`). The conversational rhythm is enhanced, not flat-stretched. Avoid centering at huge widths — keeps reading line ~70 ch. | Panel stays narrow (~360px) regardless of window width — the panel is iframe-bounded by its extension point. |

The panel's width is bounded by the host frame's panel slot; no responsive change is meaningful inside the panel beyond what the panel's own iframe gives it.

| Default | `colorScheme="primary"`, `variant="default"`, small (Blok's `sm` size if available; otherwise default) — reads as a calm informational pill |
| Hover (interactive) | When the pill is clickable (drawer trigger), pointer cursor; ring on focus per below |
| Focus (interactive) | `ring-2 ring-ring ring-offset-2` |
| Disabled | Pill renders with `colorScheme="muted"` and `aria-disabled="true"`; example: when atlas is partial and counters are framed as upper-bound estimates |
| Loading | Replaced by `@blok/skeleton h-4 w-16 rounded-full` |

#### Loading narration line (custom `<NarrationLine />`)

| State | Visual |
|---|---|
| Active (any phase) | `text-foreground text-base`, `aria-live="polite"`; updates every 200–750ms as `ScanProgress` ticks |
| Active + progress underline | Below the text, a 200px-wide track of `bg-muted` with a fill of `bg-primary`; `width: ${progress}%; transition: width 200ms linear;` |
| Stalled (>5s no progress) | Same line softly cross-fades to the slow-connection copy; underline animates a slow indeterminate sweep (`animation: pulse 1.5s ease-in-out infinite;`) using token `bg-primary/40` (or `bg-muted` if alpha tokens unavailable) |
| Cancel pending | After Stop click, narration locks to "Stopping…" for ≤500ms; then transitions to cancel-result view |
| Hidden | Atlas state ≠ `scanning`; the component unmounts |

#### Drawer (`@blok/sheet`-backed `<UsageDrawer>`)

| State | Visual |
|---|---|
| Closed | Hidden; `aria-hidden="true"` on its host element |
| Opening | Blok's default sheet slide animation (right-anchored, ~250ms) |
| Open | `bg-card`, full-height, header sentence at top (`text-lg`, `font-semibold`), two `@blok/collapsible` groups in the body |
| Loading (drawer launched before atlas resolves) | Header sentence renders with skeleton pill; body shows three skeleton rows |
| Error in detail fetch | Replace body with `@blok/error-states` |
| Empty (rare — counter says >0 but list is empty post-filter) | `@blok/empty-states` — *"No matching pages."* |

### 4.3 Accessibility notes (per NFR-4)

- **Keyboard navigation (NFR-4.1)** — Widget tab order: greeting heading (focusable for screen-reader anchoring? no — `tabindex="-1"` and addressed via heading semantics) → search input → scope select → "Refresh atlas" → first result card's "Show pages" link → … → footer affordance. Panel tab order: top-summary heading → renderings cards' counters → datasource section heading → datasource cards' counters → footer affordance. Inside the drawer: focus trap (Blok's `Sheet` does this); first focusable is the close button; Escape closes.
- **`aria-live` (NFR-4.2)** — `<NarrationLine />` renders inside an element with `aria-live="polite" aria-atomic="true"`. The phase transitions and counters are announced as text. Pulse-based content (the running narration) is rate-limited to ≤ once per 1.5s for screen-reader users (matches AC-4.2 and avoids jitter).
- **No color-only signals (NFR-4.3)** — Missing datasource: icon + word "Missing" + warning border. Partial atlas: text "Partial" + badge. Error state: icon + sentence. Skipped pages: text + count. Disambiguation suffix: badge with `colorScheme="muted"` is *secondary* — the text suffix `· abc1234` is the primary signal.
- **Display-name disambiguation (FR-9 / accessibility)** — When two renderings share a display name, both rows render the suffix (`· <last-7>`); both have `aria-label="<displayName> rendering, ID ending in <last-7>"`. Tooltips expose the full ID via `aria-describedby`, but the tooltip is **not the only** carrier — the badge text is.
- **Focus visible** — All focus rings use `ring-ring` token; never relying on `outline` color alone — ring + offset is consistent across surfaces.
- **`ⓘ Direct bindings only` affordance (per ADR-0006 / FR-8.3)** — Implemented as a `<button>` (not a passive icon) so it is keyboard-reachable and screen-reader-announced as *"Information: Direct bindings only. Click to read more."* The tooltip text from FR-8.3 is also rendered into a visually-hidden `<span>` so AT users get the explanation even if they cannot trigger hover/focus.
- **Reduced motion** — Honor `prefers-reduced-motion: reduce`. The progress underline drops the indeterminate pulse and instead becomes static; the narration line no longer fades, only swaps.
- **Contrast** — All counter-pill `colorScheme="primary"` foreground/background pairs verified for AA (≥4.5:1 for body, ≥3:1 for ≥18px) — Blok tokens are designed to pass; QA verifies both light and dark.
- **RTL** — All margins use logical properties (`ms-*`, `me-*`); the drawer anchors to the inline-end edge automatically per Blok's RTL pattern. Narration line is left/right-anchored via `text-start`.

## 5. Responsiveness and accessibility

### 5.1 Breakpoints

The Marketplace iframe constrains widths more than typical web pages. Test priorities (per Blok `theming.md` § "Iframe-specific theming"):

| Breakpoint | Behavior |
|---|---|
| **800px** | Widget: greeting heading shrinks to `text-3xl`; search input full-width; cards stack with reduced inner padding (`p-4`); pill counters wrap onto a second line if sentence is long. Panel: cards condense; rendering name and counter pill stack inside the card. Drawer: full-width sheet (`@blok/sheet` mobile mode). |
| **1024px** (default Cloud Portal iframe) | Widget: greeting `text-4xl`; search ~640px centered; result rail `max-w-3xl mx-auto`. Panel: as designed; cards have a two-column meta-row. |
| **1280px** | Widget: same as 1024 — we deliberately do **not** widen the reading column; the editorial rhythm depends on a ~70ch line. The result rail is `max-w-3xl mx-auto`. |
| **1440px+** | Widget: `text-5xl` heading; rail still `max-w-3xl mx-auto` — surface gains side margins, not column width. Panel: still narrow per its iframe slot; no change. |

### 5.2 A11y requirements (summary)

- WCAG 2.2 AA target — verified against Blok semantic tokens (light + dark) by QA.
- Keyboard-only navigation across all flows (search → cards → drawers → cancel → re-scan) with no traps outside the drawer's intentional trap.
- `aria-live="polite"` on the loading narration; `aria-atomic="true"` so each update reads as a complete sentence.
- `aria-expanded` on every collapsible toggle; `aria-controls` linking to the panel ID.
- `aria-pressed` on the scope picker if it renders as a button-row; if it's `@blok/select`, the role is automatic.
- All interactive icons (`ⓘ`, `?`, warning triangle) are `<button>` or `<a>` — never bare `<svg>` with click handlers.
- Color contrast (counter pill, narration line, missing-datasource warning, focus ring) meets AA in both themes.
- Reduced-motion: progress underline becomes static; narration line swaps without fade; collapsibles open instantly.
- RTL: tested with `dir="rtl"`; logical properties (`ms-*`, `me-*`) only; drawer anchors to inline-end automatically.

## 6. Open questions and handoff

### Open questions (for `/architect` follow-up or `/task-breakdown` resolution)

- **OQ-V3-1** — Does Blok's Nova preset expose a `bg-warning` token for the missing-datasource border? Architecture § 2.5 notes Badge intent goes via `colorScheme` and the catalog mentions `colorScheme="warning"` exists; verify at scaffold time. **Fallback:** if no warning token, use `border-destructive` (more alarming than ideal but still semantic — never hand-pick hex).
- **OQ-V3-2** — Is `@blok/search-input` shipped in the Marketplace quickstart, or do we add `@blok/input` and compose a leading-icon search field ourselves? Verify at scaffold; both compose the same conversational hero — no design impact, just a one-line task-breakdown decision.
- **OQ-V3-3** — Cards-not-table on the widget at scale: PRD does not specify an upper bound on distinct rendering count per tenant. **A-V3-2** assumes virtualization or windowing keeps NFR-1.2 (≤50ms filter) intact. The Lead Developer must pick a windowing strategy in § 4c-3 (e.g., `react-window` is too heavy; consider chunked render via `requestIdleCallback` for the first ~200 cards then on-demand). Open for Lead Developer's call; design accommodates either.
- **OQ-V3-4** — Narration tone localization. The conversational copy is English-only in v1. If the Cloud Portal locale is non-English, do we degrade to a simpler progress phrase (e.g., the phase name + counter) until copy is translated? Recommend yes for v1; revisit in Phase 2.
- **OQ-V3-5** — When the `Distract me` toggle returns in Phase 2 (per ADR-0011 escape hatch), does it offer a "less narration / more visual" mode? v3's narration *is* the distraction — making it optional would defeat the variant. Document for Phase 2 ADR.
- **OQ-V3-6** — Reduced-motion edge: when `prefers-reduced-motion: reduce`, the progress underline becomes static and the narration swaps without fade. But the narration *text changes* are themselves motion-adjacent (content shifts). Confirm WCAG's interpretation at QA — the consensus is text changes are fine; only animations need suppression.
- **OQ-V3-7** — Skipped-pages sub-drawer placement: in v3 it opens from a "see why" link inside the freshness sentence ("3 pages skipped — see why"). Should it stack inside the same `<UsageDrawer>` host (then close the parent) or open in a parallel drawer? Recommend stack-and-replace (cheaper, less visual chrome); revisit if usability testing flags it.
- **OQ-V3-8** — Card hover treatment in dark mode: `bg-muted/50` may be too subtle on the darker `bg-card`. QA visual-regression tests both themes; if dark hover is invisible, fall back to `border-primary/30` instead of an alpha background.
