## 1. Design context & assumptions

### Audience

- **Primary:** content editors mid-task in Sitecore Pages — see PRD § 4. They are decision-makers in a pre-publish / pre-delete moment. They want to *act with confidence*, not browse the data.
- **Secondary read-along:** information architects and developers running cleanup audits. The widget table's information density rewards them without harming the editor.
- **Anti-persona:** the text-only editor (PRD § 4) — the panel collapses cleanly out of their way; the widget is opt-in at install.

### Goals and success criteria (mapped to PRD)

- **G1, G2, G3** — answer *"what else uses this?"* fast on both surfaces. v2 answers it with **explicit, scannable counters at every level** — no graphs, no maps, no decoration that does not carry data.
- **G4 / S6** ("felt fast") — bearable wait. v2 reframes the wait as a *CI-style status bar* at the top of the surface, the way a developer reads a build pipeline: a thin segmented progress strip, a numerical readout, and a cancel affordance. The wait *looks like work being done*, not like a brand moment.
- **G5** — zero-infra. The visual choices (CSS-only animations, no `<canvas>`, no WebGL, no images beyond Blok graphics) keep the bundle thin and 60fps-achievable on a low-spec editor laptop (DoD-6, AR-8).

### Constraints

- **Brand & design system:** Blok (Sitecore design system) — `@blok/*` shadcn registry, semantic tokens via registry `theme.json`. Per ADR-0009 there is **no escape hatch** to Tailwind utilities for color, no shadcn-vanilla, no Material, no custom fonts. v2 stays inside the Blok token system end-to-end.
- **Architecture (locked):** two iframe entries (`/widget` + `/panel`) — ADR-0004; in-memory atlas only — ADR-0003; pull-only — PRD § 9.6; direct bindings only — ADR-0006; rendering identity by definition ID — ADR-0005; branded loading per ADR-0011 (this variant interprets "branded" as **operator-grade status**, not flourish).
- **Iframe-constrained viewports** — per `blok/theming.md` § "Iframe-specific theming," the panel surface is narrow (~360–440 px) and the widget surface is wider but still rarely full-viewport. Designs verified at 360 / 800 / 1024 / 1440 px widths.
- **A11y bar:** WCAG 2.2 AA target. Keyboard nav on the widget table (NFR-4.1), `aria-live` on scan progress (NFR-4.2), no color-only signals (NFR-4.3), 4.5:1 contrast on text and 3:1 on UI components — all handled by Blok semantic tokens out of the box; we add only the structural markup.
- **Light + dark parity** — both flip via Blok's `.dark` class on `<html>`; no hex codes; no per-mode design branch (per `blok/theming.md` § "Dark mode").

### Reference / inspiration (researched, adapted — not copied)

- **Linear** — its issue table for tight vertical rhythm, mono numerals on counters, and the "everything is a list with status chips" visual posture. Adapted: column anatomy, density.
- **Datadog dashboards** — for the *KPI strip* pattern (a horizontal row of bold counters at the top of a workspace) and the segmented status-bar pattern (multi-phase progress with active-phase pulse). Adapted: the bottom-rail KPI strip on the widget; the loading status bar on both surfaces.
- **Vercel build logs** — the "Build · Deploy · Run" segmented progress with a numerical readout. Adapted: the "Sites · Pages · Components" three-segment status bar.
- **GitHub Actions run UI** — for the cancelable, phase-pulsing run header with an unobtrusive cancel button on the far right. Adapted: the cancel-as-ghost-button pattern at the right edge of the status bar.
- **Sitecore's own Cloud Portal dashboards** — the visual baseline. Blok semantic tokens carry that DNA; v2 leans into it instead of away from it.

### Answers to clarifying questions (taken from PRD/architect open questions)

- **OQ-2 (loading visual):** "branded animation" — v2 interprets this as a **branded status console**, *not* a flourish, *not* a generative bloom. The branding comes from Blok tokens, mono numerals, the Sitecore badge pulse, and rigorous information hygiene. (Answered in § 2 / § 3 / § 4.)
- **OQ-3 (mid-scan page switch):** keep the global scan running; only the active-page row block re-paints. v2 makes this visible — the per-page row stack is its own bordered card; the rest of the panel does not flicker.
- **OQ-4 (scope change):** always re-scan from scratch. v2 surfaces the previous result in a *muted "stale" state* until the new scan lands; the status bar shows the new scan in flight without hiding the data.
- **OQ-7 / OQ-11 (pulse instrument):** v2 uses a **single-question, dismissible inline footer pulse** ("Did this help? thumbs up / down · optional one-line note"), styled like an admin-console footer toast — fits the operator metaphor.

### Assumptions made

- **A-V2-1:** Editors will not be alienated by information density; they prefer "everything visible, mono, scannable" over "warm, flowing, decorative." Validation: post-launch S1/S6 pulse vs. v1 (Atmosphere) and v3 baselines.
- **A-V2-2:** Mono numerals carry brand authority for counters and impact metrics. If Blok ships a mono variant in `theme.json`, we use it; otherwise we use `font-feature-settings: "tnum" 1, "lnum" 1` on Blok's default sans (tabular numerals — clean alignment without a separate font). § 4 documents the exact fallback.
- **A-V2-3:** A horizontal segmented status bar is *more* trustworthy than a centerpiece animation for users in this jobs-to-be-done. The wait is the work; the bar is the camera on the work.
- **A-V2-4:** "Refresh atlas" is the operator's primary control. It is hoisted to a persistent button in the surface header — never buried in a menu.
- **A-V2-5:** Editors rarely need more than 6–8 columns visible at once; v2 widget table prioritizes 5 columns at the iframe-default width and reveals 2 more (last-seen, datasource count) at ≥1024 px.

## 2. Design direction

### Visual direction — "Console Operator"

**One-line north star:** *the atlas is a control panel; the editor is at the helm; counters are how decisions get made.*

Information-dense, mono-numerical, action-first. Strong neutral surfaces from Blok (`bg-background`, `bg-card`, `bg-muted`). KPI numbers are bold, mono-tabular, large — they are the *first* thing the eye lands on. Decoration is minimal; every pixel earns its place by carrying data. Color is functional only:

| Role | Blok semantic token (Tailwind utility) | Used for |
|------|----------------------------------------|----------|
| Neutral surface | `bg-background`, `bg-card` | All canvases |
| Subtle surface | `bg-muted`, `text-muted-foreground` | Empty / pre-scan placeholders, unselected toggles, secondary metadata |
| Primary action / active counters | `bg-primary`, `text-primary-foreground` | "Refresh atlas" button, active phase in status bar, focused row chip |
| Success / completed | text + border via `border-success` / `text-success` (Blok semantic token; falls back to `text-foreground` if not exposed by `theme.json` — verify at install) | Completed scan phase chunk, "Scan completed N min ago" |
| Warning / stale / missing | `border-warning`, `text-warning`, `bg-warning/10` | Stale freshness ribbon (>15 min), "missing datasource" badge (AC-2.5), "skipped pages — view" link |
| Destructive / forbidden | `border-destructive`, `text-destructive`, `bg-destructive/10` | Permission-denied page chips in the skipped sub-drawer (FR-7.2 reason `'forbidden'`); critical scan errors |
| Accent / informational | `text-accent`, `bg-accent/10` | Info popovers — e.g. "ⓘ Direct bindings only" (FR-8.3) |

**No invented tokens. No hex.** If a Blok semantic isn't shipped (e.g. `success` not in the registry-bundled `theme.json`), the design degrades gracefully to `text-foreground` with the *icon* carrying the meaning (per NFR-4.3 — color is never the only signal).

### Layout and structure

**Both surfaces share a 4-zone vertical anatomy** — different proportions, same metaphor:

```
┌────────────────────────────────────────────────────┐
│  Zone 1 — STATUS BAR (always visible)              │  ← branded loading lives here
│  Last-scanned · scope picker · Refresh atlas       │     when scanning
├────────────────────────────────────────────────────┤
│  Zone 2 — ⓘ DISCLOSURE STRIP (compact, sticky)     │
│  "Direct bindings only" affordance + skipped link  │
├────────────────────────────────────────────────────┤
│  Zone 3 — PRIMARY DATA SURFACE (fills available)   │
│  Widget: search-first table                        │
│  Panel:  current-page rendering stack + ds group   │
├────────────────────────────────────────────────────┤
│  Zone 4 — KPI RAIL (sticky bottom — widget only)   │
│  Total renderings · Total datasources · Skipped    │
└────────────────────────────────────────────────────┘
```

The panel suppresses Zone 4 (no horizontal room); the widget keeps it always.

### Key components and patterns

- **Status bar (Zone 1)** — composed from Blok primitives only. **Not** a Blok composite; v2's branding lives here. Implemented as a horizontal flex container using `@blok/card` (already installed) as the chrome, plus `@blok/select` for scope, `@blok/button` for Refresh, and a custom `<ScanStatusBar />` segmented progress strip (described in § 4) that swaps in when state is `scanning`. The component reuses ADR-0011's `LoadingVisualizerProps` — same interface, different skin.
- **KPI rail (Zone 4)** — a sticky-bottom Blok `@blok/card` containing 3–4 KPI cells. Each cell: tiny uppercase label (Blok `text-muted-foreground text-xs uppercase tracking-wide`) above a large mono-tabular number (`text-2xl font-semibold tabular-nums`) — same styling pattern as Datadog's metric tiles, expressed in Blok tokens.
- **Widget data table** — `@blok/table` (must be installed post-quickstart per `components.md`'s gotcha list) wrapped in `@blok/scroll-area`. Tight row height (compact density variant), mono-tabular numerals on every count column. Column anatomy in § 3.
- **Panel rendering row** — composed from `@blok/card` + a custom `<CounterRow />` pattern: a *bold mono-tabular counter* on the left (e.g. "12") + small label ("other pages") + click-through chevron on the right. Implementation detail in § 4.
- **Drawer (shared by both surfaces, US-1.5 / US-2.3)** — `@blok/sheet` anchored right (widget) / sliding from the side (panel). Inside the sheet, two stacked sections separated by `@blok/separator` — "Direct rendering usage" and "Via datasource" — each with a `@blok/scroll-area`-wrapped page list. No tabs (per ADR — UX consistency across surfaces); two visible sections with anchor-linked headers.
- **Scope picker** — `@blok/select` with disabled / hidden states per AC-3.4 / AC-3.5 / AC-3.6.
- **Freshness ribbon** — `@blok/alert` with `variant="default"` when fresh, `variant="warning"` semantic when ≥15 min (per AC-5.2). One-click `@blok/button variant="ghost" size="sm"` "Re-scan" sits inline.
- **"ⓘ Direct bindings only" affordance** — `@blok/badge` + `@blok/tooltip` (or `@blok/popover` if richer copy is needed). Always visible; never collapsed (per FR-8.3 / ADR-0006 forbidden list).
- **Counters** — `@blok/badge` with `colorScheme="primary"` for active counts, `colorScheme="warning"` for missing-datasource (per architecture § 2.5 + Nova preset note).
- **Empty / error states** — `@blok/empty-states` and `@blok/error-states` per architecture § 2.5 / `marketplace-integration.md`.
- **Skeletons** — `@blok/skeleton` for pre-scan counter placeholders on the panel rendering rows. Mono-numeral typography is preserved in the skeleton (the skeleton block is the same width as the eventual digit), so the layout does not jump on resolve.

## 3. Key screens and flows

Two surfaces × five canonical states each = 10 documented screens. All screens use the 4-zone anatomy from § 2.

### Surface A — Dashboard Widget (`/widget`, mounted at `xmc:dashboardblocks`)

#### Screen W1 — Cold scan (first mount in tab)

**Zone 1 (status bar):** the segmented `<ScanStatusBar />` is the dominant element. Three chunks left-to-right: **Sites** · **Pages** · **Components**. The active chunk pulses (`bg-primary`, animated opacity 0.6→1.0 over 1.6s, `prefers-reduced-motion` respected — fades to a static fill). Completed chunks fill solid `bg-success` (or `bg-primary` fallback). Pending chunks are `bg-muted`. To the right of the bar: the numerical readout — `Pages 47 / 312 · 14s elapsed` (mono-tabular). Far right: the cancel button — `@blok/button variant="ghost" size="sm"` with an X icon and the label "Cancel" visible at ≥1024 px, icon-only at narrower widths. Above the bar, in `text-muted-foreground text-sm`: "Scanning tenant…"

**Zone 2:** the "ⓘ Direct bindings only" badge already shows — it does not wait for the scan.

**Zone 3:** an empty `@blok/skeleton`-filled table preview with 6 placeholder rows. Each row keeps the column anatomy so column widths do not jump on data arrival. Above the table, `<SearchInput />` is *disabled* with helper text "Search will activate when scan completes."

**Zone 4 (KPI rail):** all three KPIs show `—` in `text-muted-foreground`. Labels visible: TOTAL RENDERINGS · TOTAL DATASOURCES · SKIPPED.

Cancel flow: clicking cancel → `aria-live` announces "Cancellation requested." → status bar transitions to a *frozen* state with a "Canceled — partial results" badge → table populates with whatever was indexed so far → empty-state at top: `<EmptyStates>` with copy "Partial results · N pages of M scanned. Re-scan?" and a primary `@blok/button` "Re-scan."

#### Screen W2 — Completed (warm)

**Zone 1:** status bar collapses to a one-line **freshness ribbon** — `Last scanned 11:23 · whole tenant (3 collections, 8 sites, 312 pages)` on the left; scope picker (`@blok/select`) and **Refresh atlas** primary button on the right.

**Zone 2:** "ⓘ Direct bindings only" + (if any) `Skipped 3 pages — view` link in `text-warning underline`.

**Zone 3 — the data table:**

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  [search input ──────────────────]   density: [compact ●] [comfortable ○]   │
├─────────────────────────────────────────────────────────────────────────────┤
│  Rendering              │ Total ↓ │ Pages │ Mini-bar (pages%) │ Datasources │
├─────────────────────────────────────────────────────────────────────────────┤
│  HeroV2                 │     124 │   118 │ ████████████░░░░  │           7 │
│  CardGrid               │      87 │    72 │ █████████░░░░░░░  │          14 │
│  Hero  · 73fa12b        │      36 │    34 │ ████░░░░░░░░░░░░  │           1 │  ← collision suffix
│  Hero  · a04ee31        │      18 │    18 │ ██░░░░░░░░░░░░░░  │           0 │
│  (unknown rendering)    │       9 │     7 │ █░░░░░░░░░░░░░░░  │           — │  ← AR-9 group
│  ...                                                                         │
└─────────────────────────────────────────────────────────────────────────────┘
```

- Columns: **Rendering** (display name + collision suffix per FR-9 — `· <last-7-of-id>`), **Total** (mono-tabular, sortable, default sort desc), **Pages** (distinct), **Mini-bar** (CSS-only horizontal bar — `bg-muted` track, `bg-primary` fill — width = `pages / max(pages)` capped at 100%; never animated after scan; this is the "data ink" replacement for a chart), **Datasources** (count).
- Rows are `cursor-pointer`, full-row click opens drawer (US-1.5). Keyboard: arrow keys move focus row (NFR-4.1), `Enter` opens drawer, `Esc` closes.
- Last-seen-update column hidden at <1024 px to preserve density; visible at ≥1024 px.
- Density toggle (compact / comfortable) is `@blok/toggle-group` if installed; falls back to a custom segmented `<button role="radio">` group per `components.md` (toggle-group not in quickstart). Compact is default — operator preference.

**Zone 4 (KPI rail):**

```
┌──────────────────────────────────────────────────────────────────────┐
│  TOTAL RENDERINGS    TOTAL DATASOURCES    PAGES SCANNED    SKIPPED   │
│        124                  312                  309           3     │
└──────────────────────────────────────────────────────────────────────┘
```

Numbers in `text-2xl font-semibold tabular-nums`. Labels in `text-xs uppercase tracking-wide text-muted-foreground`. Tap on "SKIPPED" cell → drawer with FR-7.2 reason breakdown.

#### Screen W3 — Drawer open (rendering selected)

The right-anchored `@blok/sheet` slides in. Header: rendering display name (large, `text-lg font-semibold`) + the `· <last-7-of-id>` suffix in `text-sm text-muted-foreground` for collisions + a small ID-copy button (`@blok/button variant="ghost" size="icon"` with a copy icon — gated by ADR-0005 § "Editor abstraction line": IDs appear only in tooltips and copyable badges, never as primary identifiers). Body, in two stacked sections separated by `@blok/separator`:

1. **Direct rendering usage** (heading) → page list. Each page row: page name (left), site name (small muted), open icon (right). Hovering reveals `sitePath`. Click → `pages.context` mutate (FR-6.1).
2. **Via datasource** (heading) → for each unique datasource bound by this rendering (set per ADR-0005), a sub-card with: datasource display name + (small muted) item-display path + "+N pages" counter that opens the datasource drawer in-place.

Footer: the `@blok/button variant="ghost"` "Close" + an inline keyboard hint `Esc` rendered with `@blok/kbd`.

#### Screen W4 — Empty result / no shared usage

Per PRD § 11.3: `<EmptyStates>` (Blok primitive) — large neutral icon (Blok graphic), heading "Every component is unique to a page," body "This tenant has no shared renderings. Datasources are still indexed below."

The KPI rail still shows real numbers; only the table is empty.

#### Screen W5 — Error / no context

`<ErrorStates>` (Blok primitive) with copy from PRD § 11.3 ("Atlas needs a tenant connection — please reload the dashboard."). No retry button per OQ-A6. Status bar collapses to a single-line error chip.

### Surface B — Page Context Panel (`/panel`, mounted at `xmc:pages:context-panel`)

#### Screen P1 — Cold scan + active page resolved

Panel is narrow; the 4-zone anatomy compresses but stays.

**Zone 1 (status bar — narrow):** the segmented `<ScanStatusBar />` collapses its readout to two lines on narrow viewports — first line is the bar itself (full width), second line is `Pages 47 / 312 · 14s · Cancel`. Scope picker is hidden behind a kebab menu (`@blok/dropdown-menu`) at <420 px to save horizontal space; visible inline at ≥420 px.

**Zone 2:** "ⓘ Direct bindings only" — single line, abbreviates to just "ⓘ" with full text in tooltip on extreme narrow.

**Zone 3 — the rendering stack (active page first paint):**

The active-page rendering list paints **immediately** from the panel's own `pagesGetComponentsOnPage` call (FR-3.3 + architecture OQ-A5 — fetch issued before global scan). Each row is a `<CounterRow />`:

```
┌──────────────────────────────────────────────────┐
│   12   HeroV2                                    │
│  ─────  other pages                          → │
│  pages                                           │
├──────────────────────────────────────────────────┤
│  ···   CardGrid                                  │
│        — calculating cross-page count            │  ← skeleton state
└──────────────────────────────────────────────────┘
```

- Big mono-tabular counter on the left (`text-3xl font-bold tabular-nums text-primary` for ≥1, `text-muted-foreground` for 0).
- Below the counter, the unit label `pages` in `text-xs text-muted-foreground`.
- To the right, the rendering display name (`text-base font-medium`) and a tiny secondary line: "datasource: HeroData-X" (or `— no datasource`) in `text-xs text-muted-foreground`.
- Far right, a `→` chevron (`@blok/icon` chevron-right) indicating click-through.
- Counter shows `···` `@blok/skeleton` shimmer while the global scan is running for the count it depends on (AC-2.2). Skeleton is the same width as a 3-digit number to prevent layout shift.

**Below the rendering stack — Datasource impact group** (AC-2.4):

```
─────────────────────────────────────────────────
  Datasource impact
─────────────────────────────────────────────────
   42   HeroData-Main                            →
  pages cross-tenant
─────────────────────────────────────────────────
    ⚠   PromoBlock-Q3                            →
  missing  referenced by 7 pages
─────────────────────────────────────────────────
```

- Same `<CounterRow />` pattern.
- Missing datasources (AC-2.5) replace the count with a `⚠` glyph (`@blok/icon` alert-triangle) in `text-warning`, label `missing` in `text-xs text-warning`, with a small badge "referenced by 7 pages" — the warning is *reinforced by text* per NFR-4.3.

**Zone 4:** suppressed on the panel.

#### Screen P2 — Page-switch (panel-resident, scan still running)

When the editor navigates to a different page (via `pages.context` subscription — architecture § 5.8, D10), only Zone 3 re-paints. The status bar (Zone 1) and the disclosure strip (Zone 2) stay still. A subtle 250 ms cross-fade on the rendering stack signals the swap; CSS-only.

#### Screen P3 — Drawer open (counter clicked)

`@blok/sheet` slides from the side opposite the host. Same anatomy as the widget drawer (W3) — direct usage + via-datasource sections, page rows with open-page click-through.

#### Screen P4 — Empty / single-page rendering

If a rendering on the active page has 0 other-page usage, its counter is `0` in `text-muted-foreground` and the row is `aria-disabled="true"` for click-through (no drawer to open). Tooltip on hover: "Only used on this page."

#### Screen P5 — No tenant context

Same `<ErrorStates>` treatment as W5.

### Critical user flow — pre-publish impact check (the headline)

1. Editor on a page → opens panel → P1 paints active-page rendering stack instantly (≤1s, NFR-1.3 reuse).
2. Status bar shows `Pages 47/312 · 14s elapsed` — bearable because the bar is *honest* about progress.
3. Counters resolve as global scan completes — operator sees "12" beside HeroV2.
4. Editor clicks the "12" → P3 drawer → page list → click-through to one of those pages → makes the change with confidence.

### Responsiveness summary

| Width | Surface | Layout adjustment |
|-------|---------|-------------------|
| <360 px | Panel | Status bar wraps to 2 lines; scope picker behind kebab menu; counters stay big; rendering stack is the only scrolling region (`@blok/scroll-area`). |
| 360–800 px | Panel | Default panel layout (above). |
| 800–1024 px | Widget | Table hides "Last seen" column; KPI rail shows 3 KPIs (Skipped collapses into table header link). |
| ≥1024 px | Widget | Full table + 4-KPI rail. |
| ≥1440 px | Widget | Status bar adds a "Total elapsed during last scan" muted suffix. |

(Per `blok/theming.md` § "Iframe-specific theming," 800/1024/1440 are the explicit test points.)

### 4.2 — Custom components built on top of Blok primitives

These are *composed* of Blok primitives + Blok semantic tokens only. None of them adds raw hex, custom font families, or non-Blok dependencies.

#### `<ScanStatusBar />`

The branded loading visual (ADR-0011 surface). Implements `LoadingVisualizerProps` exactly as ADR-0011 specifies.

```ts
type ScanStatusBarProps = {
  phase: 'sites' | 'pages' | 'components';
  progress: { current: number; total: number };
  elapsedMs: number;
  onCancel: () => void;
};
```

**Markup composition** (all classes are Tailwind utilities resolving to Blok semantic tokens via `theme.json`):

- Outer `<div role="status" aria-live="polite" aria-atomic="true">` — Sites · Pages · Components live region.
- Three segmented `<div>`s in a flex row, each `flex-1 h-1.5 rounded-full`. State styling:
  - **Pending:** `bg-muted`.
  - **Active:** `bg-primary` with a CSS-only opacity pulse (`@keyframes pulse-active { 0%, 100% { opacity: 0.6 } 50% { opacity: 1 } }` 1.6s ease-in-out infinite). `prefers-reduced-motion: reduce` collapses the keyframes to a static `opacity: 1` fill.
  - **Completed:** `bg-success` (semantic; falls back to `bg-primary` if `success` is not in `theme.json`).
- Numerical readout to the right of the bar: `<span class="font-mono tabular-nums text-sm text-muted-foreground">Pages 47 / 312 · 14s elapsed</span>`. Mono uses Blok's mono variant if shipped (verify in `theme.json`); otherwise system mono via `font-family: ui-monospace, SFMono-Regular, …` — registered as `font-mono` in Tailwind config from Blok's preset. **Documented assumption (A-V2-2):** if Blok ships no mono variant, we additionally enable `font-feature-settings: "tnum" 1, "lnum" 1` on the surface root so the default sans uses tabular numerals. This keeps numerals aligned without a font swap.
- Cancel button: `@blok/button variant="ghost" size="sm"` with `@blok/icon name="x"` left of label "Cancel". `aria-label="Cancel scan"`.
- 60fps target met by using only `transform` and `opacity` (no width animation, no layout thrash) — per AR-8.

**States:** `pending` (all three pending — only on `idle` before first scan; rare), `scanning-sites`, `scanning-pages`, `scanning-components`, `completed`, `canceled` (frozen partial bar with a `Canceled` chip), `error` (single-line error chip replacing the bar).

#### `<KpiRail />`

Sticky-bottom row, widget only.

```ts
type KpiCellProps = {
  label: string;            // uppercase, e.g. "TOTAL RENDERINGS"
  value: number | '—';      // renders mono tabular
  emphasize?: boolean;      // primary color for the active filter cell
  onClick?: () => void;     // only for SKIPPED → opens skipped sub-drawer
};
```

Markup: a `@blok/card` containing a `flex justify-around` of `<KpiCell>`s. Cell:

```tsx
<button class="flex flex-col items-start px-4 py-3" aria-label={`${label}: ${value}`}>
  <span class="text-xs uppercase tracking-wide text-muted-foreground">{label}</span>
  <span class={cn('text-2xl font-semibold tabular-nums',
                  emphasize ? 'text-primary' : 'text-foreground')}>
    {value === '—' ? '—' : value.toLocaleString()}
  </span>
</button>
```

States: `idle` (`—`), `loading` (skeleton block at value width), `populated`, `populated-with-skipped` (SKIPPED cell becomes clickable + warning-tinted), `populated-empty` (zero values stay legible — `0` not `—`).

#### `<CounterRow />`

The panel's primary information unit. Composed of `@blok/card` + custom flex layout.

```ts
type CounterRowProps = {
  count: number | 'missing' | 'loading';
  unitLabel: string;          // 'pages' | 'cross-tenant'
  primary: string;            // rendering display name OR datasource display name
  secondary?: string;         // datasource ref OR site/path
  badge?: { text: string; tone: 'warning' | 'destructive' | 'accent' };
  onClick?: () => void;       // opens drawer; undefined for self-only counts
};
```

States: `loading` (skeleton in the counter slot), `default`, `zero` (counter `0` muted, row `aria-disabled`), `missing` (warning glyph + reinforcing text), `focused` (keyboard focus ring via `ring-ring`), `hovered` (`bg-muted/50` row tint).

Counter typography:

- `loading` → `@blok/skeleton` block, 3ch wide.
- numeric `≥1` → `text-3xl font-bold tabular-nums text-primary`.
- `0` → `text-3xl font-bold tabular-nums text-muted-foreground`.
- `missing` → `<Icon name="alert-triangle" />` in `text-warning`, with the word `missing` immediately below in `text-xs uppercase tracking-wide text-warning`.

#### `<DataDrawer />`

Built from `@blok/sheet`. `side="right"` for widget; `side` adapts to host frame for panel (verify host-frame contract — OQ-5 / OQ-A3). Two-section anatomy enforced via `@blok/separator`. Header has the rendering/datasource title + ID-copy button; footer has Close + `@blok/kbd` Esc hint. Body uses `@blok/scroll-area` for the page list.

**Page row inside drawer:**

```
[ icon ] Page Display Name                              [ → ]
         site • path/under/site
```

`text-sm font-medium` for page name, `text-xs text-muted-foreground` for site+path. Click → `pages.context` mutate (FR-6.1).

#### `<RenderingNameCell />`

Per architecture § 2.5. Composes `@blok/badge` (for the `· <last-7-of-id>` suffix) + `@blok/tooltip` (for the full ID).

#### `<FreshnessRibbon />`

Composes `@blok/alert` + `@blok/button variant="ghost" size="sm"` (Re-scan). Default tone matches `default`; ≥15 min stale switches to `warning` semantic + the word `Stale?` in `text-warning` next to the timestamp (text reinforces color per NFR-4.3).

### 4.4 — Accessibility notes

- **Focus management.**
  - Tab order on widget: status bar → search input → density toggle → table (rows are arrow-key-navigable, not tab-stops; first row is a tab stop) → drawer when open (focus trapped) → KPI rail.
  - Tab order on panel: status bar → scope picker → rendering rows (one tab stop, arrow keys move within) → datasource impact rows (same) → drawer.
- **Live regions.** `<ScanStatusBar />` carries `aria-live="polite" aria-atomic="true"`. Phase transitions ("Sites complete. Scanning pages.") announced once per transition. Per-page progress is **not** announced (would flood the SR queue) — only phase transitions and final summary.
- **Color-only signals avoided.** Stale = warning-tinted ribbon + the word `Stale?`. Missing datasource = warning glyph + the word `missing`. Forbidden in skipped sub-drawer = destructive-tinted chip + the word `forbidden`. (NFR-4.3.)
- **Contrast.** Blok semantic tokens already meet 4.5:1 for text and 3:1 for UI components in both light and dark. Mono numerals on neutral surfaces are 7:1+ on default Blok palettes. We do not adjust.
- **Motion.** `prefers-reduced-motion: reduce` collapses the active-phase pulse to a static fill, the cross-fade on panel page-switch to an instant swap, and the drawer slide to an instant open. All purely CSS-mediated.
- **RTL.** All horizontal layout uses logical properties (`me-*`, `ms-*`, `text-start`) per `blok/theming.md` § RTL. The status bar segment order swaps in RTL (Components on the left in RTL, mirroring the Sites-Pages-Components reading order).
- **Screen reader names.**
  - KPI cells: `aria-label="Total renderings: 124"` etc.
  - Mini-bar in widget: `aria-hidden="true"` — the data is already conveyed by adjacent number columns; the bar is decorative for sighted users.
  - Counter on `<CounterRow />`: counter + label combined as `aria-label="12 other pages use HeroV2"`.

### 4.5 — Density toggle fallback (no `@blok/toggle-group` in quickstart)

Per `components.md` "toggle-group not in quickstart": if installing `@blok/toggle-group` is rejected during § 4c-3 review, the fallback is a native button group:

```tsx
<div role="radiogroup" aria-label="Table density">
  <button role="radio" aria-checked={density === 'compact'}
          class={cn(density === 'compact'
                    ? 'bg-primary text-primary-foreground'
                    : 'bg-muted text-muted-foreground',
                    'px-3 py-1 text-sm rounded-l-md focus-visible:ring-2 ring-ring')}>
    Compact
  </button>
  <button role="radio" aria-checked={density === 'comfortable'}
          class={cn(density === 'comfortable'
                    ? 'bg-primary text-primary-foreground'
                    : 'bg-muted text-muted-foreground',
                    'px-3 py-1 text-sm rounded-r-md focus-visible:ring-2 ring-ring')}>
    Comfortable
  </button>
</div>
```

This is exactly the pattern `components.md` recommends for the `toggle-group`-not-installed case.

### 4.6 — Mono numerals — final policy

- **Preferred:** Blok ships a mono variant in `theme.json` → register it as `font-mono` in Tailwind config; use `font-mono tabular-nums` on counters and the status-bar readout.
- **Fallback (most likely path):** Blok does not ship a mono variant → register system mono (`ui-monospace, SFMono-Regular, Menlo, Consolas, monospace`) as `font-mono`. Additionally apply `font-feature-settings: "tnum" 1, "lnum" 1` to the Blok default sans on counters. This way:
  - The **numeric column alignment** (the operator-grade thing) works regardless.
  - The **mono aesthetic** is preserved on the status-bar readout and KPI rail when a mono is available.
- **Documented in § 6 as an open question** for verification at install time.

### 5.1 — Breakpoints

Anchored on `blok/theming.md` § "Iframe-specific theming" + the Blok preset breakpoint scale.

| Token | Width | Surface(s) | Behavior |
|-------|-------|------------|----------|
| (xs) | 0–360 px | Panel only | Status bar wraps 2 lines; scope picker collapses to kebab menu; counter typography drops one tier (`text-2xl` instead of `text-3xl`). Rendering rows still visible; drawer takes 90% width. |
| `sm` | 360–640 px | Panel | Default panel layout. |
| `md` | 640–800 px | Panel + small Widget | Widget table hides Mini-bar column; KPI rail still 4 cells but compressed. |

| `xl` | 1024–1440 px | Widget | Full table; KPI rail full. |
| `2xl` | ≥1440 px | Widget | Optional: status bar shows "Total elapsed during last scan" suffix. |

Tested explicitly at 360 / 800 / 1024 / 1440 px (per `blok/theming.md` Marketplace iframe rule).

### 5.2 — Accessibility requirements (WCAG 2.2 AA)

| Requirement | Implementation |
|------------|----------------|
| 1.4.3 Contrast (Minimum) | Blok semantic tokens deliver 4.5:1 on body text, 7:1+ on KPI numerals (verify with axe-core in QA per DoD). |
| 1.4.11 Non-text Contrast | Blok tokens deliver 3:1 on UI controls (buttons, focus rings, badge borders). |
| 2.1.1 Keyboard | Every interactive element reachable; arrow-key navigation on widget table; Tab/Enter/Esc canonical roles. |
| 2.4.3 Focus Order | Documented in § 4.4. |

| 4.1.3 Status Messages | `aria-live="polite"` on status bar; phase transitions announced. |

**Reduced motion:** all animations collapse to instant transitions when `prefers-reduced-motion: reduce`. Documented in § 4.4. Verified via Chrome DevTools "Emulate CSS prefers-reduced-motion" during QA.

## 6. Open questions and handoff

### Open questions for the user / next phase

- **OQ-V2-1 (typography — mono variant).** Does Blok's registry `theme.json` ship a mono font family? If yes, we lock counters and status-bar readout to it. If no, we use system mono + `tabular-nums` as the documented fallback (§ 4.6). **Resolution path:** Lead Developer at scaffold-time runs `npx shadcn@latest add https://blok.sitecore.com/r/theme.json` and inspects the resulting CSS for a `--font-mono` (or equivalent) custom property. Update § 4.6 accordingly. Architecture allows either path; design degrades gracefully.
- **OQ-V2-2 (semantic `success` / `warning` tokens).** `theme.json` may not ship `success` or `warning` semantic colors — Blok's semantic palette varies. If absent, the design degrades to icon-led, text-led indicators (per NFR-4.3 we never color-only-signal anyway). Still, we want to know whether to use `text-success` directly or fall back to `text-foreground` + check icon. **Resolution path:** Lead Developer inspects `theme.json` after install and updates the small palette table in § 2.
- **OQ-V2-3 (host-frame motion / sandbox).** Does the Marketplace iframe sandbox allow CSS `@keyframes`? (AR-10.) v2's pulse and fade are CSS-only, so we expect yes — but smoke test at first real-tenant install. **Resolution path:** real-portal smoke test per `marketplace-sdk/testing-debug.md`.
- **OQ-V2-4 (drawer side on the panel).** Per architecture OQ-5, the host-frame contract for drawer slide direction in the panel surface is unconfirmed. v2 designs with `side="right"` on widget; panel drawer side is host-determined. **Resolution path:** verify on first real-portal smoke; spec defaults to `side="left"` if the panel is host-pinned right (the default `xmc:pages:context-panel` location).
- **OQ-V2-5 (pulse instrument, OQ-7 / OQ-11 from PRD).** v2 proposes a single-question footer pulse ("Did this help? thumbs up / down · optional one-line note"). This minimizes friction and matches the operator metaphor. Approval needed before § 4c-3.
- **OQ-V2-6 (density default).** v2 defaults to **compact** in the widget. If user testing later prefers comfortable for first-time editors, the toggle persists per tab and a per-tenant default could be added in Phase 2.
- **OQ-V2-7 (KPI rail click behavior).** Currently only SKIPPED is interactive (opens skipped sub-drawer). Should TOTAL RENDERINGS / TOTAL DATASOURCES also click? v2 says no — they are summary readouts; the table is the way to drill down. Confirm during user review.
- **OQ-V2-8 (mini-bar visualization).** The mini-bar in the widget table uses `pages / max(pages)` as the bar fraction. Should it instead be `pages / totalPagesScanned` (showing tenant-share)? v2 picks the relative-to-max scaling because operators care about *relative usage hot-spots*, not absolute share. Confirm with stakeholder.
