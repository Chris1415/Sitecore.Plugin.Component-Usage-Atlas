## 1. Executive Summary

**Atlas Snapshot Export** adds a single-button download affordance to the Component Usage Atlas's two existing surfaces — Dashboard Widget and Page Context Panel — letting editors emit a portable snapshot of the atlas (full data, never filtered) in their choice of **JSON**, **CSV**, or **HTML** (the HTML doubling as the route to PDF via the browser's "Save as PDF" print dialog).

The headline use cases:

1. **Snapshot-over-time:** download a full atlas dump today, another in two weeks, diff them externally — see what got added, removed, or shifted.
2. **Hand-off / share:** send a colleague, partner agency, or stakeholder the rendering-and-page footprint of a tenant without granting them XM Cloud access.
3. **Re-use elsewhere:** import the JSON or CSV into spreadsheets, BI tools, or scripts for ad-hoc analysis the atlas itself doesn't try to do (sort by usage, compare across tenants, run cleanup recommendations).

The feature is **purely additive** on the v1 architecture (PRD-000): no new extension points, no new API scopes, no backend, no auth changes. A new module renders the in-memory atlas to one of three formats and triggers a Blob-based browser download. **Architecture 4a (iframe Mode A)** continues to apply unchanged (per ADR-0002).

This is the first PRD to extend a previously-shipped dogfood app — the atlas reached `shipped_with_caveats` on 2026-04-28, with PRD-000 covering greenfield scope. PRD-001 lives on a follow-on branch (`prd-001`) and will eventually rebase onto whatever main contains once PRD-000's PR merges.

## 2. Problem Statement

PRD-000 ships a live, in-memory atlas that answers "where is this rendering used?" and "what else uses what's on this page?" — but only **inside the iframe, only at the moment the editor looks**. The atlas dies with the tab. Three pains follow from that constraint:

1. **No way to compare across time.** An editor who scans on 2026-05-03 and again on 2026-05-17 has no way to see what changed. The most-asked question after "where is this used?" is "**did anything change since I last looked?**" — and the v1 atlas can't answer it because nothing persists.
2. **No way to share without re-running.** A solution architect reviewing a tenant for a redesign, a manager scoping a deprecation campaign, an external consultant — none of them can be handed "the atlas as of last Tuesday." They have to install the app and re-scan, or the editor has to walk them through it screen-by-screen.
3. **No way to take the data anywhere else.** Editors who want to sort by "rarely-used renderings," cross-reference renderings against a content-catalog spreadsheet, or feed the data into a cleanup PR template have to copy-paste row-by-row. The data is structurally available in the iframe but escape-hatch-locked.

The PRD-000 thesis was *"the atlas is a check-before-action tool, not a system of record"* (R10, IS-9, OS-7). That thesis is unchanged. Snapshot Export does **not** make the atlas a system of record — it lets editors **manufacture their own records, on demand, when they care**. Persistence stays out (OS-7); ad-hoc snapshots come in.

### Pain quantification — assumptions, not data

Same honest framing as PRD-000: no interview data behind these. Validation hooks are built in (see § 3 success metrics).

- **A-PAIN-1 (PRD-001):** At least one in three editors who completes a scan would download the result if the affordance existed. Source: hypothesis from product instinct — three discovery touchpoints in PRD-000 surfaced "I want to share this with X" comments. Validation: post-launch S7 download adoption rate.
- **A-PAIN-2 (PRD-001):** "Snapshot over time" is the dominant use case (over "share a single point-in-time view"). Source: user explicitly framed the feature ask as "see changes over time" first. Validation: download-frequency-per-editor over 30 days. If editors download once and never again, A-PAIN-2 is wrong and we've built a one-shot export tool, not a longitudinal one.

### Competitive landscape

The PRD-000 § 2 framing stands. No paid Marketplace app at this writing (2026-05-03) offers atlas-style export. Differentiation if a competitor surfaces:

- Free, no infrastructure, no add-on cost.
- Three formats — structured (JSON, CSV) **and** human-readable (HTML→PDF) — without paying for a separate report builder.
- One-click from the same surfaces editors already use; no separate reporting tool to learn.

## 4. Target Users

### Primary: **Content Editors** (carried unchanged from PRD-000 § 4)

Same persona. Same posture. The editor mid-task, mid-scan, who wants to walk away with a record of "what does this tenant look like right now?"

New behaviors PRD-001 enables:
- Download a snapshot before a deletion campaign; re-download after; compare externally.
- Send the JSON to a developer who's writing a migration script.
- Print the HTML to PDF and email the result to a stakeholder.

### Secondary

- **Information architects** — the export turns the atlas into an actual audit artifact they can attach to design reviews. PRD-000 explicitly **did not** optimize for this persona (OS-15); PRD-001 incidentally serves them via the JSON/CSV outputs.
- **Solution architects / external consultants** — receive shared snapshots without needing portal access.
- **Developers preparing migrations** — feed JSON into refactor scripts.

### Anti-personas — explicitly excluded from optimization

- **The continuous-monitoring user** — someone expecting scheduled exports, automatic email digests, or a webhook on every scan. Not in v1; not even queued. The user explicitly framed downloads as **editor-initiated**, manual events.
- **The integrator who wants to script downloads** — there is no programmatic API for downloads in v1. The download is a UI-button-only affordance; SDK integrators who want raw atlas data should use the Marketplace SDK directly against the same agent endpoints the atlas uses.

### Persona claims — labeled as assumptions

- **A-PERS-1 (PRD-001)** — *"editors will return to the atlas to take a fresh snapshot for diffing"* — this is the load-bearing A-PAIN-2. Validation: S9.
- **A-PERS-2 (PRD-001)** — *"non-editor stakeholders who receive shared snapshots will find HTML-with-print-stylesheet sufficient"* — assumption; if stakeholders push back wanting a "proper PDF," revisit PDF-A (client-side PDF library) or PDF-via-server in Phase 2.

## 6. User Stories

### US-1 — Widget atlas snapshot

> **As** a content editor on the Dashboard Widget,
> **I want** to download the full atlas as a JSON, CSV, or HTML file,
> **so that** I can compare it against a snapshot from two weeks ago, share it with a colleague, or feed it into another tool.

**Acceptance criteria:**
- AC-1.1 The widget header includes a single Download button (Blok-styled, icon + label) to the right of the existing "Refresh atlas" action. Visible **whenever a completed atlas is in memory** — including during a refresh-in-progress (PRD-000 IS-15 full re-scan) when a previous completed atlas is still cached. In that refresh-in-progress state the button **stays enabled** and exports the **previous** atlas with `is_partial: false`, the **previous** `scan_timestamp`, and the previous `totals`. The button is disabled only when **no completed atlas has ever been loaded** in this tab session, with tooltip "No data — start a scan first."
- AC-1.2 Clicking Download opens a Blok-styled menu with three options: **JSON**, **CSV**, **HTML**. Each option carries a one-line description ("Full data, machine-readable" / "Lite data, spreadsheet-friendly" / "Lite data, printable / shareable").
- AC-1.3 Selecting a format triggers a browser download of a single file. The button is disabled (with a spinner) for the duration of construction; re-enables after the download completes or fails.
- AC-1.4 Filename follows IS-11. Tenant name comes from the resolved tenant context (`application.context.tenantName` if available; **falls back to `tenant-<last-7-of-tenantId>`** — see § 9.4 for the canonical fallback rule applied uniformly across the filename builder).
- AC-1.5 The export reflects the **full underlying atlas** for the loaded scope. Search-box terms, sort columns, and any other view filters do **not** affect the export.
- AC-1.6 If the atlas was canceled-with-partial-results (PRD-000 AC-1.2), the export proceeds with what's loaded and the JSON top-level header sets `is_partial: true` with `pages_scanned`, `pages_total`, and `cancel_reason`. The HTML summary surfaces a visible "Partial scan — N of M pages" badge.
- AC-1.7 If construction fails (Blob too large, sandbox blocks the download, etc.), a Blok-styled toast surfaces the failure with a "Try again" affordance and a "Try a different format" suggestion. Telemetry records `export.fail` with `errorCode`.
- AC-1.8 Keyboard navigation: Download button is reachable via Tab; menu opens on Enter or Space; format options navigable via arrow keys; Escape closes the menu.

### US-2 — Panel current-page snapshot

> **As** a content editor with the Page Context Panel open,
> **I want** to download a snapshot of the current page's rendering footprint with cross-tenant counters,
> **so that** I can hand someone a record of what's on this page and how shared each component is, without giving them XM Cloud access.

**Acceptance criteria:**
- AC-2.1 The panel header includes a Download button — same Blok styling and behavior as the widget's. Visible once the panel has loaded the current page's renderings AND a tenant scan has **previously resolved** (cross-tenant counters available — even if a refresh is currently in flight, per AC-1.1's same rule). Disabled with tooltip "Loading current page…" while the panel's per-page rendering list is still resolving for the active page; disabled with "No data — start a scan first" if no completed tenant scan has ever been loaded in this tab session.
- AC-2.2 Format options identical to AC-1.2.
- AC-2.3 Filename follows IS-11 — `atlas-<tenant>-panel-<pageNameSlug>-<timestamp>.<ext>`. Page name slug uses the page's display name (slugified per § 9.4); falls back to `page-<last-7-of-pageId>` if display name unavailable.
- AC-2.4 The export contains the current page's full rendering list. JSON includes per-rendering: rendering ID, display name, parameters, bound datasource (ID + display name + path if available), cross-tenant counter, **and** the array of other pages sharing the rendering and the array of other pages sharing the datasource. CSV is lite (per IS-8). HTML is lite + summary header.
- AC-2.5 If the panel is open on a page that contains zero renderings, the export still produces a valid file (empty rendering list, populated header, IS-18 empty-state contract).
- AC-2.6 Telemetry, error handling, keyboard nav: same as US-1.
- AC-2.7 Mid-scan navigation: if the editor switches pages between download click and Blob construction, the export captures the page state **at the moment the button was clicked** (not the new page). This is implemented via FR-5.3 — the panel surface clones its `surfaceContext` (`pageId`, `pageName`, `sitePath`, `language`, plus the panel's per-rendering snapshot) at click time and passes it as a parameter to the construction function. The construction function (FR-1.1) **does not re-read panel state from any singleton or React context** — its only inputs are the atlas state, the format, and the click-time `surfaceContext` clone. Construction completes for the cloned page state; the panel re-renders for the new page in parallel.

### US-3 — Format-driven readability

> **As** an editor sending a snapshot to a non-technical stakeholder,
> **I want** the HTML version to print to a clean PDF without manual cleanup,
> **so that** the recipient gets a professional-looking artifact in one click on my end and one click on theirs.

**Acceptance criteria:**
- AC-3.1 The HTML export carries an embedded `<style media="print">` stylesheet that hides interactive chrome (none in v1, but the rule stands) and renders the summary header + table in a print-friendly layout (no unnecessary backgrounds, readable font sizes at A4 / Letter).
- AC-3.2 The HTML export is a single self-contained file (inlined CSS, no external assets, no JS, no remote fonts). No network calls when opened.
- AC-3.3 The HTML export is valid HTML5 and renders identically across the latest two stable versions of Chromium-based browsers, Firefox, and Safari. ("Identically" = visually equivalent; pixel-exact not required.)
- AC-3.4 Print preview in Chromium (the canonical XM Cloud editor browser per PRD-000 NFR-7.3) produces a clean PDF in ≤2 manual steps from the editor's perspective: open file, print, save as PDF.

### US-4 — Schema stability for diffing

> **As** an editor or analyst diffing two atlas snapshots taken weeks apart,
> **I want** the JSON shape and CSV column order to be stable across snapshots,
> **so that** a `diff` or `git diff` between the two files surfaces real content changes, not noise from reordered keys or columns.

**Acceptance criteria:**
- AC-4.1 JSON: top-level keys appear in a documented, stable order. Arrays are sorted deterministically (renderings by ID ascending; pages by ID ascending; datasources by ID ascending). No timestamp other than `scan_timestamp` and `export_timestamp` appears anywhere in the body — those are the only two fields that legitimately change between two snapshots of an unchanged atlas.
- AC-4.2 CSV: column order is fixed and documented. Header row is always present. Rows sorted deterministically (matching JSON array order).
- AC-4.3 `atlas_export_schema_version: 1` is present at the top level of every JSON export and noted in the HTML summary header (small footer-style text). Schema version bumps follow a documented breaking-change rule.
- AC-4.4 Re-running an export against an unchanged atlas produces files that differ **only** in `export_timestamp` (and the corresponding HTML/CSV-rendered timestamp). All other content is byte-identical.

### US-5 — Sandbox-failure observability

> **As** a developer or QA engineer,
> **I want** to know when a download silently fails because the iframe sandbox blocked the Blob mechanism,
> **so that** I can investigate, escalate to Sitecore platform, or pivot the mechanism without users silently losing data.

**Acceptance criteria:**
- AC-5.1 Download attempts log `export.attempt` with `{ surface, format, atlasSize, scopeKind }` to the in-iframe telemetry ring buffer (ADR-0013).
- AC-5.2 Successful saves log `export.success` with `durationMs`. The "save" detection uses the conventional Blob+anchor flow's resolved download promise (or whatever signal the verification spike establishes).
- AC-5.3 Failures log `export.fail` with one of `{ blob_construction_failed, sandbox_blocked_download, browser_save_canceled, unknown }`. `browser_save_canceled` (editor cancels the save dialog) is a benign event, not an error UX-wise — but it still gets logged.
- AC-5.4 The friction-log surfaces the breakdown of `export.attempt` vs `export.success` vs `export.fail` (per error code) at /test, and any `sandbox_blocked_download > 0` count is surfaced as a P0 finding requiring follow-up.

## 7. Functional Requirements

### FR-1 — Snapshot construction (shared)

- FR-1.1 The atlas-export module is a pure function over its arguments: `(atlas, scope, surface, format, surfaceContext) → Blob`. No side effects beyond Blob construction. Unit-testable without the SDK. **`surfaceContext` is the sole source of per-surface metadata** (active `pageId`, `pageName`, `sitePath`, `language`, panel-specific per-rendering snapshot when `surface: 'panel'`); it is a click-time clone passed by the caller. The construction function **does not re-read** panel state, React context, or any other singleton beyond the atlas itself.
- FR-1.2 Construction reads the `atlas` argument — which the caller resolves from the existing module-singleton atlas state (per ADR-0010) at click time. The construction function itself does **not** re-fetch from the SDK and does **not** re-read the singleton mid-construction. If a refresh-in-progress changes the singleton between click and construction, the export reflects the **click-time** atlas snapshot, not the in-flight one.
- FR-1.3 If the atlas is partial (canceled-with-partial-results), construction proceeds; the partial-flag fields propagate per AC-1.6.
- FR-1.4 Construction is synchronous from the caller's perspective but may yield to the event loop internally for large atlases (chunked array stringification) to keep the iframe responsive. Target: no main-thread block longer than 50ms.
- FR-1.5 Output Blobs use:
  - `application/json` for JSON
  - `text/csv;charset=utf-8` for CSV
  - `text/html;charset=utf-8` for HTML

### FR-2 — Format adapters

- FR-2.1 **JSON adapter** produces the schema documented in § 10.1. Indentation: 2-space. Pretty-printed for diff-friendliness.
- FR-2.2 **CSV adapter** produces the columns documented in § 10.2 with RFC 4180 quoting (double-quote any field containing comma / newline / double-quote; escape inner double-quotes by doubling). UTF-8 BOM **omitted** (modern tools handle UTF-8 natively).
- FR-2.3 **HTML adapter** produces the layout documented in § 10.3. Self-contained per AC-3.2.
- FR-2.4 Adapters share a common header builder (tenant name, surface, scope, scan timestamp, atlas totals, partial flag, schema version) — the builder is unit-testable in isolation.

### FR-3 — Download trigger

- FR-3.1 Construct a `Blob` of the appropriate MIME type, wrap with `URL.createObjectURL`, create a synthetic `<a download="<filename>" href="<objectUrl>">`, append to DOM, click programmatically, remove from DOM, revoke the object URL.
- FR-3.2 The synthetic anchor must be appended to the document (some browsers ignore download clicks on detached anchors). Removal happens after the click in a microtask.
- FR-3.3 If `Blob` construction throws (e.g. extremely large atlas exceeds browser limits), surface the error via the toast pattern in AC-1.7 and log `export.fail` with `blob_construction_failed`.
- FR-3.4 If the browser rejects the synthetic click (sandbox restriction, third-party iframe policy), the failure is detectable via a `<a>` `click` event handler wrapped in try/catch + a fallback timeout. **Verification spike at /architect** must confirm the detection mechanism actually works in the Marketplace iframe.

### FR-4 — Widget integration (W1)

- FR-4.1 Render a Download button in the widget header next to "Refresh atlas," using the existing Blok button primitives.
- FR-4.2 Wire the menu open/close state to local React state (no module singleton).
- FR-4.3 On format selection, call the shared construction function with `surface: 'widget'`, `surfaceContext: { /* widget-level metadata only — no per-row state */ }`, and the resolved scope from the scope filter (PRD-000 US-3).

### FR-5 — Panel integration (P1)

- FR-5.1 Render a Download button in the panel header next to existing actions.
- FR-5.2 Wire the menu state to local React state.
- FR-5.3 On format selection, snapshot the **current** panel context (`pageId`, `pageName`, `sitePath`, `language`) at click time (per AC-2.7); pass it as `surfaceContext` to the shared construction function with `surface: 'panel'`.
- FR-5.4 The panel surface's export pulls per-rendering data from the same in-memory indices the panel already uses to render the page list.

### FR-6 — Filename builder

- FR-6.1 Pure function: `(tenantName, surface, scopeKind, scanTimestamp, optional surfaceContext) → string`.
- FR-6.2 Slug rules per § 9.4.
- FR-6.3 Output never exceeds 200 characters (some filesystems have lower limits; be conservative). If a slug component would push the filename over, the page-name slug is truncated to 60 chars with a 7-char ID suffix appended for uniqueness.

### FR-7 — Empty-state handling

- FR-7.1 An atlas with zero renderings still produces a valid export per IS-18.
- FR-7.2 A page with zero renderings (P1 surface only) still produces a valid export per AC-2.5.

### FR-8 — Partial-scan handling

- FR-8.1 Per AC-1.6 — partial atlases export with explicit partial flags. No silent gaps.
- FR-8.2 Skipped pages (per PRD-000 FR-7.2 — `forbidden`, `timeout`, `not_found`, `network_error`, `other`) are surfaced in the JSON export under `skipped_pages` with their reason codes. CSV omits skipped pages from the data rows but includes a "skipped" footer note. HTML surfaces them as a "N pages skipped" line in the summary header.

### FR-9 — Telemetry

- FR-9.1 `export.attempt`, `export.success`, `export.fail` events written to the ring buffer per AC-5.1–5.3.
- FR-9.2 No PII beyond what PRD-000 NFR-5.3 already permits (page IDs, rendering IDs).
- FR-9.3 Events are emitted via the existing telemetry module — no new transport.

- NFR-1.3 No measurable regression to existing atlas performance metrics (PRD-000 NFR-1.1, NFR-1.2, NFR-1.3) caused by the new module's presence.

### NFR-2 — Reliability

- NFR-2.1 Construction is deterministic given a stable atlas input (per AC-4.4).
- NFR-2.2 No silent failures: every download attempt emits exactly one `export.success` or `export.fail` event.
- NFR-2.3 If the iframe sandbox blocks the download mechanism, the user sees a clear toast (not a frozen button); telemetry surfaces `sandbox_blocked_download`.

### NFR-3 — Accessibility

- NFR-4.2 No tokens, scan secrets, or auth headers leak into any export.

### NFR-5 — Maintainability

- NFR-5.1 Format adapters live in three small modules under a single `src/atlas/export/` directory. Each adapter is ≤300 LOC.
- NFR-5.2 Schema version constant lives in a single file; bumping the version touches one location.
- NFR-5.3 Tests for adapters use fixture atlases, not the SDK; test coverage targets ≥90% for the export module.

### NFR-6 — Observability (carried forward from PRD-000 NFR-6, extended)

### NFR-7 — Deployment / Architecture posture (carried unchanged)

## 9. Technical Considerations

### 9.1 — SDK surface

- **No new SDK calls.** The export reads from existing in-memory atlas indices (per ADR-0010 module singleton) populated by PRD-000's scan engine.
- **No new agent endpoints.** No `xmc.agent.*` or `xmc.sites.*` calls beyond what PRD-000 already does.
- The `application.context` lookup for tenant name reuses the existing context resolution (PRD-000 FR-1.1). Verification spike: confirm `application.context.tenantName` exists. If unavailable, fall back to **`tenant-<last-7-of-tenantId>`** per the canonical rule in § 9.4 (consistent with AC-1.4). The fallback is the same string everywhere — filename, JSON `tenant.tenant_name` (which surfaces `null` separately to indicate the fallback was used), and HTML/CSV header.

### 9.2 — Architecture posture

- Architecture 4a (iframe Mode A) — unchanged. ADR-0002 still applies.
- The export module is a leaf module; nothing else in the atlas depends on it.
- State: read-only against the existing module singleton (ADR-0010). The export module does not write atlas state.

### 9.3 — Stack

- TypeScript. No new heavy dependencies.
- No client-side PDF library (per IS-10, OS-3).
- Format adapters: pure TypeScript, ≤300 LOC each.

### 9.4 — Slug rules (filenames and scope labels)

- Lowercase ASCII.
- Replace whitespace and punctuation with `-`.
- Collapse runs of `-`.
- Trim leading/trailing `-`.
- Truncate page-name slugs to 60 chars max, append `-<last-7-of-pageId>` if truncated.
- Display-name collisions for collections: `<displayNameSlug>-<last-7-of-collectionId>` per IS-17.
- Tenant name slug: same rules; **falls back to `tenant-<last-7-of-tenantId>`** if tenant name unavailable. This exact string is also used for the JSON/HTML/CSV header presentation when tenant name is missing — the fallback is a single, canonical token applied uniformly across filename and body.
- ISO timestamp in filenames: compact form `YYYYMMDDTHHMMSSZ` (no colons/dashes — filesystem-safe). Body content uses canonical ISO-8601 with separators.

### 9.5 — Iframe sandbox download mechanism

- **Mechanism:** `Blob` + `URL.createObjectURL` + synthetic anchor + programmatic click + revoke (per FR-3.1).
- **Risk:** Sitecore Marketplace iframe historically allows this, but it has not been verified for the specific extension points the atlas uses (`xmc:dashboardblocks`, `xmc:pages:context-panel`).
- **Verification spike at /architect:** required. /architect must produce a small standalone test harness (or extend an existing one) that triggers a Blob download from each of the two extension points in a real Cloud Portal install, and reports back: success / browser-prompts-and-saves / silently-blocked / explicitly-errors. If "silently blocked," the spike must propose a fallback (e.g. opening the export in a new tab via `window.open` with a data URL — though data URLs hit size limits at ~2MB in some browsers).
- **Anti-pattern:** do not implement the export and "test it later." The verification spike runs before the implementation does, because a failed mechanism invalidates the entire feature.

### 9.6 — Schema versioning

- The `atlas_export_schema_version` field starts at `1`. Future versions:
  - Adding a top-level field: minor change, **schema version unchanged** (but a `schema_minor` field may be introduced if needed).
  - Removing a field, renaming a field, or changing field semantics: **major bump** to schema_version.
  - Reordering fields: never (deterministic order is contractual per AC-4.1 / IS-14).

### 9.7 — Data sources from PRD-000 atlas state

Confirmed mapping:

| Export field | Source in atlas state |
|--------------|------------------------|
| Rendering ID + display name | `renderingIndex.<key>` (ID is the key); display name resolved from existing rendering metadata cache |
| Datasource ID + display name | `datasourceIndex.<key>` (ID is the key); display name resolved via `useDatasourceNames` hook (S22/S23) |
| Page metadata (id, name, path, language, site) | `PageRef` shape per PRD-000 § 10.1 |
| Cross-tenant counters | `renderingUsage.pages.length` and `datasourceUsage.pages.length` |
| Tenant context | `application.context.*` |
| Scan timestamp | `Atlas.scannedAt` |
| Atlas totals | `Atlas.totals` |
| Skipped pages | `Atlas.skipped[]` |

No new data needs to be plumbed into the atlas state.

## 10. Data Model Overview

### 10.1 — JSON schema (canonical)

```ts
type AtlasExportV1 = {
  // Top-level metadata block — stable order
  atlas_export_schema_version: 1;
  surface: 'widget' | 'panel';
  format: 'json';                    // self-identifying for tooling
  exported_at: string;               // ISO-8601 — click time
  scan_timestamp: string;            // ISO-8601 — when scan completed
  is_partial: boolean;
  partial_info?: {
    pages_scanned: number;
    pages_total: number;
    cancel_reason: 'user_canceled' | 'timeout' | 'error';
  };
  tenant: {
    tenant_id: string;
    tenant_name: string | null;      // null when context doesn't expose name
  };
  scope: {
    kind: 'all-collections' | 'collection';
    collection_id?: string;
    collection_name?: string;
  };
  languages_scanned: string[];       // dedup, ASC sort; e.g. ["en"] or ["de","en"]
  totals: {
    sites: number;
    pages: number;
    renderings: number;
    datasources: number;
  };
  skipped_pages: Array<{
    page_id: string;
    reason: 'forbidden' | 'timeout' | 'not_found' | 'network_error' | 'other';
  }>;

  // Body — surface-specific
  body: WidgetBody | PanelBody;
};

type WidgetBody = {
  renderings: Array<{
    rendering_id: string;
    display_name: string;
    total_usages: number;
    distinct_pages: number;
    last_seen_update: string | null;
    pages: PageRef[];                // full array, sorted by page_id asc
    datasources: Array<{
      datasource_id: string;
      display_name: string | null;
      pages: PageRef[];              // pages where this rendering binds this datasource
    }>;
  }>;
};

type PanelBody = {
  page: PageRef;                     // current page metadata
  renderings: Array<{
    rendering_id: string;
    display_name: string;
    parameters: Record<string, string> | null;
    cross_tenant_page_count: number;
    cross_tenant_pages: PageRef[];   // other pages sharing this rendering
    bound_datasource: {
      datasource_id: string;
      display_name: string | null;
      path: string | null;
      cross_tenant_page_count: number;
      cross_tenant_pages: PageRef[]; // other pages sharing this datasource
    } | null;
  }>;
};

type PageRef = {
  page_id: string;
  page_name: string;
  site_path: string;
  site_id: string;
  site_name: string;
  language: string;
  collection_id?: string;
};
```

No IndexedDB, no localStorage, no service worker cache. Snapshots are constructed on demand and exist only in the user's downloads folder. PRD-000 ADR-0003 unchanged.

## 11. UX Considerations

### 11.1 — Surface treatments

- **Widget header:** existing controls left-justified (search box, scope filter, refresh atlas, last-scanned timestamp). New Download button right-justified, just before any overflow menu. Icon: download arrow. Label: "Download" (visible at desktop widths; icon-only with tooltip at narrow widths).
- **Panel header:** narrower; Download button placed inline with refresh action. Same icon. Label hidden behind tooltip due to width constraints — acceptable per Blok header conventions for narrow surfaces.

### 11.2 — Format picker menu

- Blok dropdown menu pattern.
- Three options:
  - **JSON** — small subtitle: "Full data, machine-readable"
  - **CSV** — small subtitle: "Lite data, spreadsheet-friendly"
  - **HTML** — small subtitle: "Lite data, printable / shareable"
- Each option has the format extension shown after the label as muted text: `JSON .json`, `CSV .csv`, `HTML .html`.
- Menu closes on selection, Escape, click-outside.

### 11.3 — Loading + disabled states

- Button disabled during scan with tooltip: "Scan in progress — finish the scan to download."
- Button disabled during construction (rare; <1.5s) with inline spinner replacing the icon. Tooltip: "Preparing your download…"
- Button disabled during cancel-with-no-results state with tooltip: "No data — start a scan first."

### 11.4 — Toasts

- **Success:** Blok toast variant `success`, short — "Downloaded `<filename>`." Auto-dismisses in 4s.
- **Failure:** Blok toast variant `error`, sticky until dismissed — "Couldn't download. <reason>. Try a different format or retry." Includes "Retry" + "Dismiss" actions.
- Sandbox-blocked failure adds a third action: "Copy to clipboard (JSON)" — emergency fallback that copies the JSON to clipboard if the Blob download mechanism fails. This is an opportunistic fallback, not a standalone user-facing feature.

### 11.5 — Visual identity

- Download icon matches the existing iconography in the atlas surfaces (consistent stroke weight, ~16px).
- Menu styling matches the existing scope-filter dropdown in the widget.
- HTML print output uses the editor-friendly typography (11pt sans-serif body, slightly smaller monospace for IDs in headers).

### 11.6 — Accessibility checklist

- Per NFR-3 — Tab reachability, arrow-key menu nav, Escape to close, descriptive option labels, screen-reader-friendly disabled states.
- Tooltips do **not** carry information that's not also accessible to screen readers (every disabled state has an aria-describedby pointing to the same explanation).

### 11.7 — Empty state copy

- Empty atlas (zero renderings): the export still proceeds; the resulting file is a valid empty export. The success toast says "Downloaded `<filename>` — empty atlas."
- Page with zero renderings (P1 surface): same — exports proceed, toast acknowledges.

## 14. Open Questions

These are explicitly open and to be resolved in `/architect` or design phases. The PRD does not commit to answers here.

- **OQ-1 (architect):** Does the Sitecore Marketplace iframe sandbox permit the standard `Blob + <a download>` pattern in **both** `xmc:dashboardblocks` and `xmc:pages:context-panel`? Verification spike — see § 9.5.
- **OQ-2 (architect):** Does `application.context` expose `tenantName` directly, or only `tenantId`? Affects filename UX (R2).
- **OQ-3 (architect):** What's the correct fallback if Blob download is sandbox-blocked? Current candidates: (a) clipboard copy of JSON only (other formats lose), (b) data: URL in new tab (size-limited), (c) instruct editor to use browser dev tools (terrible UX). /architect picks one.
- **OQ-4 (UI design):** Should the format-picker menu also show a one-line **size estimate** ("JSON — Full data, ~2.3MB") for the current atlas? Could reduce surprise on large tenants. Cost: extra computation on menu open.
- **OQ-5 (UI design):** Print stylesheet typography source — match the in-app Blok tokens, or pick a generic high-readability print font? Trade-off: brand consistency vs. print-quality optimization.
- **OQ-6 (UX research):** Does the success toast linger long enough at 4s? Users may want a "show in folder" affordance — Marketplace iframe has no FS access. Document and move on.
- **OQ-7 (architect):** Should the export module live under `src/atlas/export/` (proposed) or somewhere else in the existing PRD-000 module layout? Code-organization decision; ADR likely.
- **OQ-8 (PM/UX):** Should the menu copy emphasize the format **purpose** ("For sharing," "For analysis," "For printing") instead of the format name? More approachable for non-technical editors. Cost: less precise; A/B reasonable in Phase 2.
- **OQ-9 (security):** CSV formula injection mitigation strategy — prefix suspicious fields with `'` (Excel-friendly), or reject the injection and fail-safe? Default to prefix per OWASP (R4).
- **OQ-10 (architect):** Telemetry — should `export.success` events include the file's **size** (KB/MB)? Useful for diagnosing R3; question is whether it counts as PII (almost certainly not). Default: yes, include.

## Amendment 1 — 2026-05-04 — Three-action egress (pageshot pattern)

### Trigger

T001 verification spike (the first task of `/task-breakdown` per ADR-0017) ran on 2026-05-04 against a real Cloud Portal install on a real test tenant. Outcome: **F0 (canonical `Blob + URL.createObjectURL + synthetic <a download> + click + revoke`) returned `silent_block` on BOTH extension points** (`xmc:dashboardblocks` widget and `xmc:pages:context-panel` panel). Diagnostic `console.info` instrumentation confirmed the click handler fires and `a.click()` returns synchronously, but no save dialog appears, no file lands in Downloads, and no console error is raised. The user clarified this is a **known platform-level limitation** in the Sitecore Marketplace iframe sandbox (the host frame is mounted without `allow-downloads`), not a per-app defect. The same constraint is documented independently in the sibling Pageshot product at `products/pageshot/site/next-app/components/use-open-image.ts:7-15`.

### Decision

The single-Download-button cascade originally specified by ADR-0017 (F0 → F1 fallback → F2 fallback → F3 hard-fail toast) is **superseded** by the **three-action egress pattern** mirrored from Pageshot's production-verified solution. Atlas Snapshot Export ships **three first-class user-visible actions per format**:

1. **Save** — canonical `Blob + URL.createObjectURL + synthetic <a download>` mechanism (preserved from ADR-0017 § Primary mechanism). Renders **disabled** in the current sandbox; future-proof for the day Sitecore adds `allow-downloads`.
2. **Open** — `window.open(blobUrl, '_blank', 'noopener,noreferrer')` opens the formatted artifact in a real top-level browsing context where the browser handles JSON / CSV / HTML natively. HTML → PDF route is preserved (Open in new tab → Ctrl+P → Save as PDF).
3. **Copy** — text-format-aware. JSON / CSV use `navigator.clipboard.writeText(text)`; HTML uses `navigator.clipboard.write([new ClipboardItem({ 'text/html': ..., 'text/plain': ... })])` so paste targets that understand HTML get formatted output and plain-text targets get the raw markup.

Cite **ADR-0021** (`project-planning/ADR/adr-0021-three-action-egress-pageshot-pattern.md`) for the full architectural justification.

### Acceptance criteria amendments

The following AC IDs are amended in spirit by this addendum. AC IDs whose IDs are not listed continue to apply unchanged.

- **AC-1.1, AC-2.1** — replace **"Download button"** with **"format picker + three-action cluster (Save / Open / Copy)"**. Save's enabled-state condition is unchanged (atlas loaded), but Save is rendered with the additional `disabled-pending-platform-fix` state described below — the action button is visually disabled with a tooltip while the sandbox lacks `allow-downloads`. The previous tooltips for `disabled-no-data`, `disabled-panel-loading`, and `disabled-scan-in-progress-no-prior` apply to all three actions equivalently.
- **AC-1.2, AC-2.2** — format picker still drives JSON / CSV / HTML choice. Subtitles unchanged ("Full data, machine-readable" / "Lite data, spreadsheet-friendly" / "Lite data, printable / shareable"). After format selection, the **action cluster** (three pills) is exposed.
- **AC-1.3** — replace **"format selection triggers a browser download"** with **"action selection triggers the chosen action: Save (where supported by sandbox), Open in new tab, or Copy to clipboard. The clicked action button is disabled with a per-action spinner during construction; the other two action buttons remain interactive."**.
- **AC-1.7** — failure handling shifts from a single failure toast to **per-action status taxonomies**. The blob-construction-failure path (a cross-cutting failure that prevents any action) still surfaces a Sonner error toast. Per-action blocks (`open` blocked by popup → `'blocked'`; `copy` denied by clipboard → sticky `'denied'`; `save` permanently disabled by sandbox → `'unsupported'`) surface inline status copy on the action affordance itself, NOT a toast. The toast strategy from PRD-001 § 11.4 is reshaped per ADR-0021 § Consequences / Harder.
- **AC-2.7** — unchanged in spirit (mid-navigation export captures click-time state per ADR-0016) but applies to the **action click**, not the format-pick click. The format pick alone does not freeze panel context; the click that triggers Save / Open / Copy is the click whose state is cloned via `cloneSurfaceContext`.
- **AC-5.1, AC-5.2, AC-5.3** — telemetry shifts. Each action emits its own `export.attempt`, `export.success`, or `export.fail` with `action: 'save' | 'open' | 'copy'` added to the payload. The `errorCode` union is widened with **`popup_blocked`** (Open's `window.open` returned `null`). Save's `sandbox_blocked_download` becomes the **persistent reason for the disabled state**, not a per-click error — emitted at most once per session as an informational `export.fail` when the surface mounts (so operators can compute "% of installs with `allow-downloads`" from the ring buffer), not on every Save click.

### New Functional Requirements

- **FR-10 — Save action.** Hook `useSaveExport({ blob, filename }) → { status: 'idle' | 'saving' | 'saved' | 'unsupported', save: () => void }`. Implementation matches ADR-0017 § Primary mechanism (Blob + objectURL + synthetic anchor + click + revoke with 60 s deferred revoke). At mount time, the hook MAY run a no-op feature-detection probe (`'download' in document.createElement('a')`); the probe does NOT detect Marketplace's silent_block (the API is "supported" at the document level; the sandbox swallows the result). The disabled-state policy on the action button is enforced at the **surface level** via ADR-0021's recommendation, not via runtime detection. Future-proof: when Sitecore adds `allow-downloads`, the disabled guard at the surface flips off and Save begins working with no hook code change. Cite Pageshot's `useDownloadImage` at `products/pageshot/site/next-app/components/use-download-image.ts:99-110` as the reference implementation.

- **FR-11 — Open action.** Hook `useOpenExport({ blob }) → { status: 'idle' | 'opening' | 'opened' | 'blocked', open: () => void }`. Implementation: `window.open(URL.createObjectURL(blob), '_blank', 'noopener,noreferrer')`. On `null` return value, status flips to `'blocked'` (sticky for the session) and the action affordance surfaces an inline message pointing at Copy. `URL.revokeObjectURL` is **deferred 60 seconds** so the new tab has time to read the blob (per Pageshot's pattern). Cite `products/pageshot/site/next-app/components/use-open-image.ts:62`.

- **FR-12 — Copy action.** Hook `useCopyExport({ text, mode: 'text' | 'html' }) → { available: boolean, status: 'idle' | 'copying' | 'copied' | 'denied' | 'unsupported', deniedMessage: string, copy: () => Promise<void> }`. Text path (JSON / CSV): `navigator.clipboard.writeText(text)`. HTML path: `navigator.clipboard.write([new ClipboardItem({ 'text/html': new Blob([text], { type: 'text/html' }), 'text/plain': new Blob([text], { type: 'text/plain' }) })])` so paste targets that understand HTML get formatted output and plain-text targets get raw markup. **Sticky `'denied'`** posture per Pageshot precedent: after a single rejection, status stays `'denied'` for the lifetime of the hook (no auto-revert). Cite `products/pageshot/site/next-app/components/use-copy-image.ts:128-148` — atlas's variant **diverges** in mime type only (Pageshot uses `image/png` ClipboardItem; atlas uses `text/*`).

### UX shape

The surface header (widget) and zone-2 (panel) host a **format picker** (existing PRD § 11.2 dropdown — JSON / CSV / HTML) followed by a **three-action cluster** (Save / Open / Copy pills). Save is rendered disabled with the tooltip:

> "Downloads are blocked in this iframe — use Open or Copy instead. (Save will work once Sitecore enables it.)"

Open and Copy are first-class affordances and equally visible. The PRD-000 visual posture ("Quiet Affordance" per UI design v1) is preserved — the action cluster uses the same Blok button primitives the POC used (`@blok/button` outline / ghost variants, `@blok/icon`). Visual reference for the action affordance anatomy: `products/pageshot/site/next-app/components/ActionPill.tsx` (live in production).

### What did not change

- **ADR-0002** (no backend) — both Open and Copy are pure client-side hooks; no server traffic introduced.
- **ADR-0003** (no persistence) — no IndexedDB / localStorage / service-worker cache added.
- **ADR-0010** (atlas state singleton) — export still reads at click time only; construction function is still pure.
- **ADR-0013** (telemetry in-iframe) — same ring buffer; new payload field (`action`) and new errorCode (`popup_blocked`) extend the existing union.
- **ADR-0018** (no client-side PDF library) — HTML + browser print remains the only PDF route, now invoked through Open's new tab → Ctrl+P → Save as PDF.
- **ADR-0019** (schema versioning) — `ATLAS_EXPORT_SCHEMA_VERSION` constant unchanged at `1`; no field added or renamed in the JSON schema.
- **ADR-0020** (tenant identity) — `requireTenantIdentity(ctx)` resolver unchanged; tenant fallback unchanged.
- **PRD-001 G5 ("zero new architecture surface")** — still accurate at the deployment level: same Mode A iframe, same scopes, no backend.
