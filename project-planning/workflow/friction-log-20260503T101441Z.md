## download_smoke

The T001 spike (ADR-0017) verifies the canonical iframe-download mechanism
(`Blob + URL.createObjectURL + synthetic <a download> + click + revoke`) on
both atlas extension points. Outcomes recorded as one entry per surface per
attempt.

### Attempt 1 — F0 primary mechanism (synthetic `<a download>` click)

| Field | widget | panel |
|-------|--------|-------|
| surface | `xmc:dashboardblocks` (route `/widget`) | `xmc:pages:context-panel` (route `/panel`) |
| result | **`silent_block`** | **`silent_block`** |
| browser | (TBD — user did not capture) | (TBD — user did not capture) |
| timestamp | 2026-05-04 (initial T001 deploy) | 2026-05-04 |
| console_excerpt | none | none |

**Notes.**
- Initial deploy: user reported "click does nothing" on both surfaces — no save
  dialog, no file in Downloads, no console error.
- Diagnostic patch landed (`console.info('[CUA-SPIKE] runDownloadSpike fired', …)`
  before any DOM work + after `a.click()`). Redeploy + retest: user confirmed
  BOTH `[CUA-SPIKE] runDownloadSpike fired` and `[CUA-SPIKE] a.click() returned`
  log lines fire on both surfaces, AND no save dialog / file lands in Downloads,
  AND no console error.
- Interpretation: the click handler executes, the synchronous `a.click()`
  returns without throwing, but the iframe sandbox silently swallows the
  download. This is the textbook `silent_block` outcome per ADR-0017 and
  AC-5.3 (`errorCode: 'sandbox_blocked_download'`).
- **Verdict:** `fork-to-F1` per ADR-0017's fallback hierarchy. F0 primary
  mechanism is **NOT** viable on either extension point in the current
  Cloud Portal sandbox configuration.

**Actions triggered by this outcome.**
1. F1 verification spike (next attempt below) — `window.open(objectUrl, '_blank')`
   variant.
2. iframe sandbox attribute diagnostic — log `window.frameElement?.sandbox` from
   inside the iframe at mount time so we know exactly what the host grants.
3. T002+ work remains blocked until F1/F2/F3 picks a working fallback.
4. ADR-0017 amendment will record the F0 outcome + chosen fallback before
   `/code-review` runs.

### Attempt 2 — F1 fallback (`window.open` of Blob URL)

**Not run as a per-spike verification.** User confirmed on 2026-05-04 that downloads are a known platform-level limitation in the Marketplace iframe sandbox (the host iframe omits `allow-downloads`). Rather than continue cascading single-mechanism spikes, the run forks to the **pageshot-pattern three-action egress** (ADR-0021) — Save preserved as future-proof, Open via `window.open` and Copy via `navigator.clipboard` ship as primary user-visible actions. F1 (`window.open`) inherits production-proven evidence from `products/pageshot/site/next-app/components/use-open-image.ts:62`; F2 (clipboard) inherits from `use-copy-image.ts:128`. Per-action smoke verification will run at `/test` time inside the host frame, not as a one-off T001 spike.

## Architectural fork — pageshot-pattern adoption (2026-05-04)

**Trigger.** F0 silent_block confirmed on both `xmc:dashboardblocks` and `xmc:pages:context-panel`. User reported "downloads are forbidden — known limitation in permissions" and asked: "Can we offer the download differently like we did with pageshot?"

**Investigation.** Grepped `products/pageshot/site/next-app/components/` for `download | saveAs | window\.open | clipboard`. Found three production hooks (`useDownloadImage`, `useOpenImage`, `useCopyImage`) shipping a parallel three-action UX. Pageshot's inline comments document the exact same iframe-sandbox constraint. Cited at `use-open-image.ts:7-15`.

**Decision.** Adopt the pageshot pattern verbatim, adapted for text artifacts (JSON / CSV / HTML) instead of pageshot's binary PNG. ADR-0021 captures the full architectural shift; ADR-0017 supersedes-in-part to point at ADR-0021 while preserving the Save mechanism specification.

**Downstream.** PRD-001 amended via Amendment 1; prd-minimal-001 carries a one-paragraph note in Key constraints; task breakdown surgically amended (T001 marked completed with verdict; new tasks for `useOpenExport` + `useCopyExport` + `useSaveExport` hooks; E5 surface integrations reshape to three-action cluster). UI design v1 spec carries an Amendment 1 in § 7. POC frame additions for the three-action shape deferred to `/document` polish.

**Comparison method.** PRD-000 baseline = `before_bytes` from the T002
entry above (1,288,631 raw — captured before `npx shadcn add @blok/sonner`
ran). No gzipped baseline was recorded at PRD-000 ship time, so the
gzipped delta uses the 0.35 ratio T002 cited. The file-level sum at
after_T041 is reported alongside as a more precise figure.

**Cap status — FINDING (escalation to /ship gate).**

`smoke_outcomes.bundle_delta.verdict: 'fail'` — the cumulative gzipped
delta (~54 KB by the 0.35-ratio estimate; ~38 KB if we credit only the
post-T002 export feature additions) exceeds the PRD-001 NFR-1.4 / DoD-5
cap of **≤20 KB gzipped** by between 1.9× (export-feature-only view)
and 2.7× (full cumulative view).

Drivers (top chunks at `.next/static/chunks/`):
- `396f90e8b7ccb31f.js` — 224 636 B raw / 70 148 B gz (largest single chunk;
  likely the consolidated framework + Blok primitives + shadcn/ui bundle).
- `98d2211b5807ff56.js` / `ef15a90add60198b.js` — 180 562 B raw each
  (~57 KB gz each; route-level bundles for `/widget` and `/panel`).
- `c5eec2964a31a88d.css` — 69 883 B raw / 12 815 B gz (Tailwind v4 +
  Blok tokens; not the export feature directly, but Tailwind's per-route
  utility set scales with component count).

The export feature's *own* contribution is hard to attribute precisely
without a per-feature bundle analyzer pass — the largest plausible
attributions are the new `core/atlas/export/**` module tree (≈10 source
files), the three `useSaveExport` / `useOpenExport` / `useCopyExport`
hooks, the format-picker + three-action egress UI, and the `sonner`
toast primitive (already accounted for in T002's ~11–13 KB).

**Recommendation for /ship.**
1. Run a precise gzipped-baseline measurement against the PRD-000
   ship commit before declaring this a hard fail — the 0.35 ratio
   may overstate the gzipped baseline (Tailwind v4 CSS gzips
   substantially better than 35 %, dragging the ratio down).
2. If the cap is genuinely breached after a precise re-baseline, raise
   an /architect amendment to either (a) revise NFR-1.4 (the egress
   action surface is inherently larger than the original "single
   download button" and the cap may have been set before ADR-0021's
   three-action expansion), or (b) defer the heaviest format adapter
   (HTML's inlined print stylesheet is the largest single export-side
   asset) to a Phase 2.
3. Surface this in the /ship completion summary as the headline risk
   item; do NOT silently cross the cap.

This finding is the primary escalation from the T042–T050 verification
batch.
