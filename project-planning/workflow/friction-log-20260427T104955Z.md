### 2026-04-27 — T008 / M1 verification — `notFound()` returns HTTP 200, not 404

ADR-0014 mandates "root route → `notFound()`" and T008 expected output states "`http://localhost:3000/` returns Next.js 404." Verified in browser: `notFound()` IS firing server-side (RSC payload contains `NEXT_HTTP_ERROR_FALLBACK;404` and the standard `"404: This page could not be found"` markup). However:

- HTTP wire-level status is **200**, not 404. Next 16 App Router renders `notFound()` content **inside the root layout**, which means `<MarketplaceProvider>` wraps the rendered 404. While the provider is in `loading === true` state (the default outside the portal iframe), its loader (`<div>Attempting to connect to Sitecore Marketplace...</div>`) renders in place of children — so the user sees the loader, not the 404 page.
- A direct hit to `/_not-found` returns HTTP 404 cleanly (the standalone not-found route bypasses the layout), confirming Next is correctly configured.
- A real undefined route (`/does-not-exist`) also returns HTTP 404 normally.

Practically harmless — editors never visit `/` in the portal (they hit `/widget` or `/panel` extension routes), and a developer doing local smoke-test sees the same loader on `/widget` and `/panel` outside the iframe, so `/`'s loader is consistent. ADR-0014's intent ("root unreachable") is met at the application-semantics level even if the HTTP code isn't 404.

**Skill patch / ADR amendment suggestion:** ADR-0014 should acknowledge the layout-precedence behavior in Next 16 App Router. Either (a) amend the ADR to say "root renders the provider's standby state in standalone-localhost; portal-iframe behavior is the only one that matters" or (b) consider moving `<MarketplaceProvider>` from `app/layout.tsx` to nested `app/widget/layout.tsx` + `app/panel/layout.tsx` so root is provider-free and 404s cleanly. Option (b) contradicts T008's "Do NOT remove the root provider" guard and may break the panel↔widget switch UX, so option (a) is the safer call. Recommended: revisit at M6 / `/architect` review pass.

### 2026-04-27 — T015 — `client.query` mock pattern needed THREE-layer envelope, not two

§ 4 T015 / § 4c-6 prose described the unwrap as `result.data?.data ?? []`. The first draft of T107's mock SDK returned the hey-api success envelope directly from `client.query` — i.e. `{ data: T, error: undefined, request, response }`. Production code then read `result.data.data` (double-unwrap) but `result` was already the envelope, so `result.data === T` and `result.data.data === undefined`. Result: every test reported `0 sites` and `0 pages` even though the fixtures had thousands. **Resolution:** mocks must return a `QueryResult`-shaped wrapper whose `.data` slot carries the hey-api envelope — i.e. THREE nominal layers when looking at tests:

```ts
return {
  data: { data: <payload>, error: undefined, request, response },  // hey-api envelope inside QueryResult.data
  error: undefined, status: 'success', isLoading: false, isError: false, isSuccess: true,
  refetch: () => Promise.resolve({} as never),
};
```

This is purely a TEST-FIXTURE concern — production code is unaffected because the production `client.query` already returns the wrapper. **Skill patch suggestion:** `client.md` § 9 (typed-mock pattern) should ship a `mockQueryResult(payload)` helper that the codifies the wrapper-of-envelope-of-payload structure so test authors don't reinvent it.

### 2026-04-27 — T015 — `unwrapOk` helper to discriminate hey-api envelopes without `as` casts

Hey-api `RequestResult` has TWO branches:

```ts
ThrowOnError === true:  { data: T;                request; response }
ThrowOnError === false: { data: T; error: undefined } | { data: undefined; error: E } & { request; response }
```

Both expose `data` but only the failure branch sets `error` to non-undefined. A naive optional-chain `result.data?.data` works at runtime but TypeScript narrows the union poorly because it can't see the `error === undefined` discriminant. **Resolution:** added a small `unwrapOk<T>(envelope)` helper in `lib/sdk/queries.ts` that checks `error !== undefined` first and returns `T | undefined`. Keeps the boundary free of `as any` / `as never`.

### 2026-04-27 — T025 — `scanning → scanning` self-edge needed for phase transitions

### 2026-04-27 — M4 / T110 / T044 — Collision detection re-uses M2's `computeCollisions` against the prop-supplied set

### 2026-04-27 — M5 / vitest setup — Radix primitives need ResizeObserver + matchMedia in jsdom

Blok's `<Sheet>` and `<ScrollArea>` wrap Radix primitives that observe their content size at mount and consult `prefers-reduced-motion`. jsdom ships neither `ResizeObserver` nor `matchMedia` by default, so the M5 drawers crash on render with `ReferenceError: ResizeObserver is not defined`. **Resolution:** added no-op stubs for both in `vitest.setup.ts` (top of file). ### 2026-04-27 — M5 / T040 — Widget surface auto-trigger uses on-mount-only effect, not state-driven

§ 4c says "On first mount with state.kind === 'idle', call triggerScan(...)". Strictly state-driven would mean a `useEffect(() => { if (state.kind === 'idle') trigger(); }, [state.kind])` — but that re-fires the trigger every time state cycles back to idle (e.g. after `resetAtlas` from a future user-initiated reset). atlas-actions has its own strict-mode guard inside `triggerScan`, so re-entry is harmless, but it surfaces as a noisy second `triggerScan` call in tests. **Resolution:** mount-only `useEffect(() => { if (idle) trigger(); }, [])` with an `eslint-disable-next-line` for the missing dep — comments explain that we depend on the bare minimum on purpose. This matches the v2 POC behaviour (one auto-scan on first paint) and keeps T064's re-mount-during-scan test honest.

### 2026-04-27 — M6 / T050 — `pages.context` subscribe-via-query callback shape needs an `as any` escape

Per `client.md` § 6a Path A the `pages.context` subscribe call goes through `client.query('pages.context', { subscribe: true, onSuccess, onError })` and returns an object with `unsubscribe()`. The current `@sitecore-marketplace-sdk/client@0.3.2` `.d.ts` for `ClientSDK.query` does not expose the subscribe overload generically — `subscribe`/`onSuccess`/`onError` aren't in the `QueryOptions` union the type system sees. **Resolution:** cast the options object as `any` at the single call site inside `panel-surface.tsx` (one `// eslint-disable-next-line @typescript-eslint/no-explicit-any`) and locally type the returned subscription as `{ unsubscribe?: () => void }`. ADR-0007 / `40-sdk-contracts.mdc` allow narrow boundary casts when the SDK shape is genuinely missing — kept it confined to the subscribe path. **Skill patch suggestion:** `marketplace-sdk/client.md` § 6a should note that the runtime accepts the subscribe-shape but the type signature has not been narrowed for it yet, and document the safe single-cast pattern so future implementations don't roll their own.

### 2026-04-27 — M6 / T050 — `react-hooks/set-state-in-effect` rejects the obvious "clear state on dependency change" pattern

In the panel's per-page-fetch effect, the natural shape is "if no active page, set components to empty array; otherwise kick off a fetch". Next 16's eslint preset flags the `setPageComponents([])` line in the no-active-page branch as an error per `react-hooks/set-state-in-effect`. **Resolution:** stop holding the empty-list invariant in state and DERIVE it instead — keep `pageComponents = { forPageId, components } | null` and compute `displayedComponents = pageComponents.forPageId === activePageId ? pageComponents.components : []`. State only changes when fetch resolves; "clearing" is a render-time projection, not an effect side-effect. Same pattern is recommended in React 19 docs. **Skill patch suggestion:** `30-tdd.mdc` and/or a new `60-react-effects.mdc` rule should pre-emptively warn implementers that "clear-on-dep-change" via setState in an effect is now a lint error in Next 16 — derive the value instead.

### 2026-04-27 — M6 / T074 — `<DebugPanel />` URL detection via useState+useEffect tripped the same effect rule

`new URLSearchParams(window.location.search).get('debug')` is a window-scoped read so it can't be done at render time directly without breaking SSR contract — the obvious pattern is `useState(false)` + `useEffect(() => setEnabled(...))`. That trips the same `react-hooks/set-state-in-effect` rule. **Resolution:** model the debug flag as an external store via `useSyncExternalStore` with a no-op subscribe (the value can only change on a hard reload anyway), so the read happens at render but stays SSR-safe via the explicit `getServerSnapshot` returning `false`. Same shape ADR-0010 mandates for atlas state. **Skill patch suggestion:** add a "useSyncExternalStore for non-React-mutating browser-only reads" idiom note to the React rules — it's becoming the canonical answer to the new lint rule.
