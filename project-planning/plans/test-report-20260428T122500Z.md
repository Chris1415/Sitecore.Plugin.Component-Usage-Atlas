### Major

#### M_NEW1 — `withBackoff` still hardcodes `surface: 'widget'` in `rate_limit_retry` telemetry events (M5 fix incomplete)

- **File and line:** `site/core/scan-config.ts:74`
- **Category:** correctness (telemetry tagging)
- **Severity:** major
- **Description:** The M5 fix from the code-review pass plumbed `surface` through the scan engine's own telemetry calls (`scan_started`, `scan_completed`, `phase_transition`, etc.). However, `withBackoff` (called inside `queryComponentsOnPage` per `lib/sdk/queries.ts:416` and exercised on every page-fan-out) still emits its `rate_limit_retry` event with `surface: 'widget'` hardcoded. The trailing comment `// re-mapped by the engine when wired` was carried over from before the M5 fix and is no longer accurate — there is no remapping. As a result, panel-initiated scans that hit rate limits will record `rate_limit_retry` events tagged as `'widget'` even when the originating surface was `'panel'`. This is the same class of bug M5 was supposed to fix.
- **Reproduction:** Mount `<PanelSurface />` while the test tenant is rate-limiting; inspect the telemetry ring buffer (`?debug=1`) — every `rate_limit_retry` entry surfaces as `surface: 'widget'`.
- **Suggested fix:** Thread `surface` through `withBackoff` (add an optional `surface?: ScanSurface` parameter; pass from `queryComponentsOnPage` which receives it from the engine via a new arg, defaulting to `'widget'` for backwards compat). Alternatively: drop the surface tag from `rate_limit_retry` events entirely (it's a per-call retry, not a per-surface KPI) and update the conformance test fixture. The first option is preferred because it keeps cross-surface telemetry analysis coherent.

#### M_NEW2 — Orphan scaffold component renders entire `appContext` JSON (latent ADR-0013 leak)

- **File and line:** `site/components/examples/built-in-auth/application-context.tsx:38-41`
- **Category:** security / housekeeping
- **Severity:** major (latent — currently unused, but lives in production source tree)
- **Description:** A scaffold-leftover component `<ApplicationContext />` renders `JSON.stringify(appContext, null, 2)` inside a `<CollapsibleContent>`. The `appContext` includes `resourceAccess[0].context.live` / `.preview` (the tenant context IDs) which ADR-0013 forbids surfacing in any UI / log / DOM. The component is currently NOT imported anywhere (Grep `application-context` returns no consumers), so the leak is dormant — but a future developer wiring up a "debug" route could revive it without realizing the ADR consequence. Same class of bug that M4 fixed in `marketplace.tsx`. The neighboring scaffold file `list-languages.tsx:50,53` also has dormant `console.log` statements that would log SDK responses if revived.
- **Suggested fix:** Delete `site/components/examples/built-in-auth/` entirely. The directory is scaffold debt unrelated to the atlas product; nothing imports it; ADR-0013 risk per file is non-trivial; and the empty-states.tsx / error-states.tsx scaffold-derived warnings are a separate cleanup that could ship alongside.

### Minor

#### m_NEW3 — Unknown-rendering virtual row may receive a spurious collision suffix

- **File and line:** `site/components/atlas/widget-table.tsx:103-118` (synthesizes virtual row); collision computed at `widget-table.tsx:138-141`
- **Category:** correctness / cosmetic
- **Severity:** minor
- **Description:** When the atlas contains 2+ unknown renderings (each with display name `"(unknown rendering)"`), `<WidgetTable />` collapses them into ONE virtual row with `renderingId = unknowns[0].renderingId`. The `collisionMap` is computed over the FULL `allRenderings` set, so it sees N unknown entries with identical `displayName` and emits a `· <last-7-of-id>` suffix for the first one's ID — which IS the virtual row's ID. The displayed virtual row therefore gets a misleading suffix like `(unknown rendering) · abc1234`, suggesting that the virtual placeholder collides with itself. The unknown-rendering test (`unknown-rendering.test.tsx`) does not assert the absence of a suffix.
- **Suggested fix:** Filter out `isUnknown` entries before passing to `computeCollisions` inside `<WidgetTable />`, OR special-case the virtual row's `<RenderingNameCell />` to not pass `collisionMap`. Add a regression test: virtual row's rendered text equals `(unknown rendering)` exactly (no suffix).

#### m_NEW4 — Layout metadata still scaffold default

- **File and line:** `site/app/layout.tsx:16-19`
- **Category:** quality / cosmetic
- **Severity:** minor
- **Description:** `metadata.title` is `"Create Next App"` and `description` is `"Generated by create next app"`. Already flagged as m3 in the prior code review; not auto-fixed. The host frame typically owns the title for Marketplace apps so this is cosmetic only.
- **Suggested fix:** Set to `{ title: "Component Usage Atlas", description: "Live tenant-wide rendering and datasource usage atlas." }`.

#### m_NEW5 — `forbiddenPageIds` Set re-created on every render of both surfaces

- **File and line:** `components/atlas/widget-surface.tsx:319-323`, `components/atlas/panel-surface.tsx:388-394`
- **Category:** performance
- **Severity:** minor
- **Description:** Already flagged as m5 in the prior code review; not auto-fixed. Inline `new Set(skipped.filter(...).map(...))` in JSX changes Set identity on every render, defeating drawer-side memoization.
- **Suggested fix:** `useMemo(() => new Set(skipped.filter(s => s.reason === 'forbidden').map(s => s.pageId)), [skipped])`.

#### m_NEW6 — `useEffect` dependency arrays use `eslint-disable-next-line` for stable refs

- **File and line:** `components/atlas/widget-surface.tsx:212`, `components/atlas/panel-surface.tsx:171, 234`
- **Category:** quality
- **Severity:** minor
- **Description:** First-mount effects (`triggerScan` + `surface_mounted`) and the `pages.context` subscribe effect use `// eslint-disable-next-line react-hooks/exhaustive-deps`. The intent is correct (we want exactly-once behavior at mount), but the disable hides the fact that `client` and `contextId` could change across re-renders without the subscription re-attaching. The route components currently resolve fresh `contextId` per render, so a context swap would produce a stale subscription. This is mitigated by the route-level remount on context-change being unlikely (the marketplace provider handshake is single-shot), but it's a sharp edge.
- **Suggested fix:** Either depend on `[client, contextId]` and re-subscribe on change (with the `subscribe` indirection eating the duplicate), or wrap the values in `useRef` and document the assumption explicitly.
