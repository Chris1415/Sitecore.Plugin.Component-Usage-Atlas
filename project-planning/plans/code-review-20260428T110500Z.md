## Critical Findings

### C1 — `as any` cast at SDK boundary violates 10-language.mdc + 40-sdk-contracts.mdc

- **File and line:** `site/components/atlas/panel-surface.tsx:187`
- **Category:** architecture (rule violation) + quality
- **Severity:** critical
- **Description:** Panel surface calls `client.query('pages.context', { subscribe: true, onSuccess, onError })` but uses `as any` to satisfy the TypeScript compiler because the typed signature on `ClientSDK.query` does not advertise the `subscribe`/`onSuccess`/`onError` shape directly for the `pages.context` key. `10-language.mdc` and the prd-minimal non-negotiables list (TypeScript-first; no `as any`) forbid this. The `eslint-disable-next-line @typescript-eslint/no-explicit-any` annotation acknowledges the violation rather than fixing it.
- **Suggested fix:** Replace `as any` with a typed local options shape that mirrors `BaseQueryOptions<T,E,P>` from `@sitecore-marketplace-sdk/client/dist/types.d.ts` and cast through `unknown` once. Move the inline ESLint disable. This narrows the `as` surface to a single typed cast against a documented interface rather than `any`.

## Major Findings

### M1 — `RenderingNameCell` recomputes collisions O(N²) on every render

- **File and line:** `site/components/atlas/rendering-name-cell.tsx:39`
- **Category:** performance
- **Severity:** major
- **Description:** Each row in the widget table mounts a `<RenderingNameCell />`, and inside the cell `computeCollisions(Array.from(allRenderings.values()))` runs over the full rendering set on EVERY render. With N rows (potentially hundreds in a real tenant) the table re-renders cost O(N²) for the collision computation alone. The collision map is a function of `allRenderings` only — it should be memoized once at the surface level (or cached on the Map identity) and passed to each cell.
- **Suggested fix:** Compute the collision map once inside `<WidgetTable />` (and once inside `<RenderingImpactList />` for the panel), wrap it in `useMemo` keyed on the renderings Map reference, and pass it down as a `collisionMap` prop. The cell becomes O(1) per row.

### M2 — Empty-atlas case shows wrong copy ("Every component is unique")

- **File and line:** `site/components/atlas/widget-table.tsx:160`
- **Category:** correctness
- **Severity:** major
- **Description:** When the rendering index is empty (`isEmptyAtlas` true), the table renders `<AtlasEmptyState mode="no-shared" />`, whose copy reads "Every component is unique to a page. This tenant has no shared renderings." That message describes a totally different state — it implies the scan worked and found unique components. For a truly empty atlas (no pages at all, or scan yielded zero renderings) the correct copy is `mode="empty-tenant"` ("No published pages — This tenant has no published pages yet."). Editor will be confused by the misleading state copy.
- **Suggested fix:** Use `mode="empty-tenant"` for the `isEmptyAtlas` branch. Keep `mode="no-shared"` reserved for when the atlas is non-empty but every rendering is a singleton (a future feature; today it's never reachable).

### M3 — `requireContextId` throws synchronously inside route component, no error boundary catches it

- **File and line:** `site/app/widget/page.tsx:20`, `site/app/panel/page.tsx:15`
- **Category:** correctness + architecture
- **Severity:** major
- **Description:** Both route components call `requireContextId(appContext)` at render time. If the provider yields a non-null `appContext` whose `resourceAccess[0].context` does not contain `live`/`preview`, the function throws `AtlasNoContextError`. There is no React error boundary in `site/app/layout.tsx` to catch it, and the surfaces' `error` state with `priorAtlas` rendering is never reached. The architecture spec § 10.1 W5/P5 says the surface should render the no-context state — but the throw happens before the surface mounts.
- **Suggested fix:** In each route component, wrap the call in try/catch and pass `contextId={null}` (or render a small no-context fallback). Better: have `requireContextId` return `string | null` and let surfaces drive the no-context state explicitly through atlas state. Minimal patch: replace the throwing call with a try/catch that renders the no-context copy.

### M4 — `console.log("appContext", res.data)` in MarketplaceProvider leaks context payload to browser console

- **File and line:** `site/components/providers/marketplace.tsx:37`
- **Category:** security
- **Severity:** major
- **Description:** The Marketplace provider logs the entire `application.context` response to the browser console on every successful initialisation. The context payload contains `resourceAccess[]` entries with tenant/context IDs that are quasi-secret (per ADR-0013 telemetry should not surface tenant identifiers). In a Mode A iframe the console may also be observable to the host frame depending on devtools attachment. ADR-0013 mandates no PII / no tenant identifiers in any telemetry surface.
- **Suggested fix:** Remove the `console.log` (or gate it behind `?debug=1` like the debug panel). The provider was inherited from the scaffold; this is a cleanup pass.

### M5 — Hardcoded `SURFACE = 'widget'` mistags telemetry events fired on panel-initiated scans

- **File and line:** `site/core/scan-engine.ts:87`
- **Category:** correctness
- **Severity:** major
- **Description:** The scan engine constant `SURFACE = 'widget' as const` is used in every telemetry `track()` call inside `runScan`. When `<PanelSurface />` triggers a scan via `triggerScan`, all subsequent `scan_started`, `scan_completed`, `scan_canceled`, `scan_error`, `phase_transition`, and `rate_limit_retry` events are mistagged as `surface: 'widget'`. The telemetry conformance test fixes `surface: 'widget'` and never exercises `surface: 'panel'`, so the bug is silent today. This breaks any later telemetry-driven analysis that distinguishes which surface initiated the scan.
- **Suggested fix:** Plumb a `surface: 'widget' | 'panel'` field into `ScanInput` and use `input.surface` (default to `'widget'`) for the telemetry tag. Update both surfaces to pass the right value through `triggerScan` / `refreshAtlas` / `setScope` (atlas-actions also threads it).

## Minor Findings

### m1 — `as never` casts in test files

- **Files:** `site/core/__tests__/atlas-freeze.test.ts:66, 82`; `site/core/__tests__/scan-engine.integration.test.ts:132`
- **Category:** quality
- **Severity:** minor
- **Description:** The TypeScript-first / no-`as never` rule is mostly a production-code rule. Tests use `as never` to test that mutating-method calls throw at runtime (e.g. `(atlas.skipped as Array<never>).push(...)`); this is a reasonable test-only shortcut. Flagged for awareness — a stricter cleanup would replace with explicit `// @ts-expect-error` annotations.

### m2 — `drawer-row.tsx` references `React.ReactElement` without importing the `React` namespace

- **File and line:** `site/components/atlas/drawer-row.tsx:51`
- **Category:** quality
- **Severity:** minor
- **Description:** Type annotation uses `React.ReactElement` while only `type ReactNode, type KeyboardEvent` are imported from 'react'. Works because `@types/react` ships an ambient `React` namespace, but it's inconsistent with the rest of the codebase (other components use `import type * as React from 'react'`).

### m3 — Layout still has scaffold's default metadata "Create Next App"

- **File and line:** `site/app/layout.tsx:16-19`
- **Category:** quality
- **Severity:** minor
- **Description:** `metadata.title` is "Create Next App" and `description` is "Generated by create next app". Cosmetic — host frame typically owns the title — but worth tightening to "Component Usage Atlas".

### m4 — `clearScanInFlight()` not called on the strict-mode guard early-return path

- **File and line:** `site/core/scan-engine.ts:158`
- **Category:** correctness
- **Severity:** minor
- **Description:** When `markScanStarting()` returns false (a scan is already in flight), `runScan` returns a no-op handle whose `donePromise` resolves immediately. The `finally { clearScanInFlight() }` in the actual scan body still fires on the in-flight scan, so this is correct — but the visual structure can mislead a reader. Add a one-line comment clarifying that the second invocation does NOT clear the in-flight flag (the original scan's finally does).

### m5 — `panel-surface.tsx` re-creates `forbiddenPageIds` Set on every render of UsageDrawer

- **File and line:** `site/components/atlas/panel-surface.tsx:367-371`, `site/components/atlas/widget-surface.tsx:319-323`
- **Category:** performance
- **Severity:** minor
- **Description:** Both surfaces compute `new Set(skipped.filter(...).map(...))` inline as a prop to `<UsageDrawer />`. The Set identity changes on every render even when `skipped` does not. Wrapping in `useMemo([skipped])` would stabilise the prop and let the drawer's downstream memoization bail out.

### m6 — `vitest.setup.ts` uses `as unknown as` to install a ResizeObserver polyfill

- **File and line:** `site/vitest.setup.ts:19`
- **Category:** quality
- **Severity:** minor
- **Description:** The cast pattern works but pollutes a test setup with unsafe casts. `globalThis.ResizeObserver = ResizeObserverStub` (with a top-of-file `declare global` shim) would be cleaner.
