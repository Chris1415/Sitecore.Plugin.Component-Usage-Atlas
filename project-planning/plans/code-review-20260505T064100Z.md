## Minor findings

### 1. `html.ts` — print stylesheet line-number citations are stale

**File:** `core/atlas/export/formats/html.ts:83, 86, 88, 90, 91`
**Category:** quality (documentation provenance)
**Severity:** minor

The HTML adapter inlines hex color values for the print stylesheet (legitimate per ADR-0021 + UI design § 4.7 — the HTML output is opened OUTSIDE the iframe so CSS-variable lookup against `globals.css` is unavailable; values must be derived). Each value carries a citation comment like `// globals.css:14 (--foreground)`. **The cited line numbers are wrong** — the actual semantic tokens live much later in `globals.css` (at lines 497, 506, 507, 511, 532). The line-number citations point at the Blok color-scale block (`--color-blue-50` etc.), not the semantic-token block they claim to source.

The hex *values* are also approximations rather than exact resolutions. Example: `#4a4a4a` cited for `--muted-foreground`, but `--muted-foreground: var(--color-blackAlpha-500) = rgba(0,0,0,0.55) ≈ #737373`. The visual difference is imperceptible on print, but the citations claim authoritative provenance the values don't actually have.

**Suggested fix:** either (a) update the line citations to point at the actual semantic-token declarations (`globals.css:497-549` block), and resolve the hex values exactly via the alpha-on-white mapping, OR (b) rewrite the citations as `// derived from Blok --muted-foreground (alpha-resolved on white background)` without specific line numbers — same provenance signal without the brittleness.

### 2. `html.ts` — three lines lack the citation comment that other lines carry

**File:** `core/atlas/export/formats/html.ts:92, 93, 94`
**Category:** quality (consistency)
**Severity:** minor

Lines 83-91 each carry an inline `// globals.css:NN (--token)` citation. Lines 92-94 (the `.empty-state`, `.if-skipped`, `footer` rules) reuse hex values from earlier rules but don't repeat the citation. Consistency with the surrounding lines means a future reader doesn't have to scroll up to figure out where these came from.

**Suggested fix:** add citation comments to lines 92-94. Cheap and improves readability; ties in with the (1) fix above.

### 3. Bundle delta — escalation from T044 (open from /implement)

**File:** N/A (planning concern)
**Category:** architecture (NFR-1.4 / DoD-5)
**Severity:** minor (deferred to /document or /ship for resolution)

T044's audit measured cumulative gzipped delta ~37.9 KB (file-summing × ratio estimate) vs the 20 KB cap in NFR-1.4. ADR-0021's three-action pattern is structurally heavier than the original single-Download concept the cap was sized for. Two paths previously surfaced: (a) re-measure with `next-bundle-analyzer` for an exact feature-attributable share, or (b) amend NFR-1.4 to reflect the post-fork architecture. **Not a code defect** — the cap measurement is approximate and the cap was set pre-fork.
