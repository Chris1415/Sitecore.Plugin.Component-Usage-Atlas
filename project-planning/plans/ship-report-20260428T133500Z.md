## 1. Summary

PRD-000 set out to ship **Component Usage Atlas** — a Sitecore Marketplace app shipping a **Dashboard Widget** (component-centric search) and a **Page Context Panel** (page-centric impact) as a tenant-wide, live, in-memory atlas of rendering and datasource usage, in Mode A only with no backend and no persistence. ## 4. Known Limitations

1. **Standalone-localhost browser hit on `/widget` or `/panel` shows the SDK loader, not the surface.** When opened outside the Cloud Portal iframe, the `MarketplaceProvider` stays in `loading === true` state and renders `<div>Attempting to connect to Sitecore Marketplace…</div>` instead of the surface. This is **by design** (Mode A iframe-only per ADR-0002 + ADR-0014); the surface only renders inside the Cloud Portal iframe. README documents the smoke-test rule: always validate inside the host frame, never via raw localhost.
2. **Root `/` returns HTTP 200, not 404, in standalone-localhost.** ADR-0014 mandates `notFound()` from the root route. Next 16 App Router renders `notFound()` content **inside the root layout**, where the `MarketplaceProvider` loader masks the 404 markup; the wire-level status stays 200. `/_not-found` and any undefined route correctly return 404. ADR-0014's intent ("root unreachable in the portal flow") is met at application semantics — editors hit `/widget`/`/panel` extension routes, never `/`. Flagged in friction log as an **ADR-0014 amendment candidate**: acknowledge layout-precedence behavior in the ADR text or move the provider to per-surface layouts (the latter contradicts T008's "do NOT remove the root provider" guard).
3. **Scan time on very-large tenants is unmeasured.** NFR-1.1 sets a <30s cold-scan budget on ~1k pages; the underlying logic + concurrency pool + backoff are in place, but actual wall-clock behavior on tenants with thousands of pages depends on XMC backend rate-limit responses and editor patience. Measurement is gated on T094.
4. **Cross-tenant counter accuracy depends on the in-memory atlas being current.** The atlas is session-scoped and dies with the tab; manual "Refresh atlas" is the only re-population mechanism in v1 (incremental refresh is OS-13). Editors who leave a tab open for hours after a peer's publish will see stale counts unless they refresh.
5. **Display-name collision suffix is `· <last-7-of-id>`.** Renderings are guaranteed unique by item ID, so the disambiguation is always correct. Edge case: if two renderings ever shared an identical display name AND the same first 25 characters of their ID, the visual suffix could feel cluttered. This is a 1-in-trillions edge case; accepted as-is.
